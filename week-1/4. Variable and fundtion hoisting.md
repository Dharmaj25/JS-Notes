# Hoisting — SDE2 Interview Reference
> Your only source of truth for this topic. Read this before every interview revision.

---

## What is Hoisting?

Before JavaScript runs a single line of your code, it does a **preparation step**.  
During this step, JS scans your entire code, finds all declarations, and **allocates memory for them**.

This "picking up declarations before running code" is called **hoisting**.

> Hoisting is NOT about code physically moving.  
> It's about JS allocating memory for declarations during the creation phase — before your code runs line by line.

---

## Why Does Hoisting Exist?

It was a design decision — not a bug.

JS was designed so that **function declarations** could be used anywhere in a file, even before they appear. This let developers write "main" logic at the top and helper functions at the bottom.

`var` hoisting was a side effect of that same mechanism.  
`let`/`const` TDZ behavior was intentionally introduced later to fix the confusion `var` caused.

---

## The Two Phases — The Foundation

Every time JS runs code (a script or a function call), it creates an **Execution Context** with two phases:

```
┌──────────────────────────────────────────────────────────────┐
│  PHASE 1 — CREATION PHASE  (before any code runs)           │
│                                                              │
│  JS scans all declarations and allocates memory:            │
│                                                              │
│  var declaration       → stored as undefined                 │
│  let/const declaration → stored but locked in TDZ ⚠️        │
│  function declaration  → entire function body stored ✅      │
│                                                              │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│  PHASE 2 — EXECUTION PHASE  (code runs line by line)        │
│                                                              │
│  Assignments happen here                                     │
│  Functions get called                                        │
│  let/const exit TDZ when their declaration line is reached  │
│  Function declaration lines are SKIPPED (already done)      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

> This two-phase process is part of the **Execution Context** — you'll go deeper into this on Day 5.

## Part 1 — Variable Hoisting

### var — hoisted and initialized to `undefined`

Only the **declaration** is hoisted. The **assignment** stays where you wrote it.

```js
console.log(a);  // undefined — no error, already in memory
var a = 5;
console.log(a);  // 5
```

What JS actually does:

```js
// Creation phase:
var a = undefined;  // ← hoisted

// Execution phase:
console.log(a);     // undefined
a = 5;              // assignment happens here
console.log(a);     // 5
```

---

### var hoists to function scope — not block scope

```js
function test() {
  console.log(x);   // undefined — hoisted to top of function

  if (false) {
    var x = 10;     // if block never runs — but hoisting still happens!
  }

  console.log(x);   // undefined — assignment never ran
}

test();
```

**Why?** Hoisting happens in the **creation phase** — before conditions are evaluated.  
The `if (false)` never executes, but `var x` is still hoisted. The answer is `undefined, undefined` — not an error.

```
Creation phase of test():
  → finds: var x
  → hoists to top of function scope → x = undefined
  
Execution phase:
  → if (false) → never runs → x never assigned
  → both logs print undefined
```

---

### let and const — hoisted but placed in TDZ

```js
console.log(b);  // ❌ ReferenceError: Cannot access 'b' before initialization
let b = 5;
```

**Proof that `let` IS hoisted** — the inner block TDZ shadow:

```js
let x = 1;
{
  console.log(x);  // ❌ ReferenceError — NOT 1!
  let x = 2;
}
```

If the inner `let x` wasn't hoisted, `console.log(x)` would print `1` from the outer scope.  
But the inner `let x` IS hoisted to the top of the block and placed in TDZ — it shadows the outer `x`.  
Accessing it before its line = ReferenceError.

---

### The two different ReferenceErrors — know both

```
"Cannot access 'x' before initialization"
→ variable EXISTS (hoisted) but is in TDZ
→ caused by let or const

"x is not defined"
→ variable does NOT exist anywhere in scope chain
→ caused by accessing a truly undeclared variable
```

---

### Variable Hoisting Summary

```
DECLARATION       HOISTED?    INITIALIZED TO    USABLE BEFORE LINE?
──────────────────────────────────────────────────────────────────────
var x = 5         ✅ Yes       undefined          ✅ Yes (returns undefined)
let x = 5         ✅ Yes       TDZ ⚠️             ❌ No (ReferenceError)
const x = 5       ✅ Yes       TDZ ⚠️             ❌ No (ReferenceError)
```

## Part 2 — Function Hoisting

JS hoists different kinds of functions very differently. This is critical.

---

### Function Declarations — fully hoisted (name + entire body)

```js
greet();  // ✅ "Hello!" — works before the declaration

function greet() {
  console.log("Hello!");
}
```

What JS does after creation phase:

```js
// Creation phase — entire function lifted:
function greet() {
  console.log("Hello!");
}

// Execution phase:
greet();  // ✅ works perfectly
```

The entire function is stored in memory during the creation phase.  
This is intentional — call your helper functions before defining them.

---

### Function Expressions with var — partially hoisted

```js
sayHi();  // ❌ TypeError: sayHi is not a function

var sayHi = function() {
  console.log("Hi!");
};
```

What JS does:

```js
// Creation phase — only var is hoisted:
var sayHi = undefined;

// Execution phase:
sayHi();              // calling undefined() → TypeError ❌
sayHi = function() {  // function assigned here
  console.log("Hi!");
};
```

The **variable** is hoisted as `undefined`. The **function body** is not hoisted.  
Calling `undefined()` gives `TypeError: sayHi is not a function`.

---

### Arrow Functions — same as function expressions

```js
sayHello();  // ❌ TypeError: sayHello is not a function

var sayHello = () => {
  console.log("Hello!");
};
```

Arrow functions are always assigned to variables.  
They follow the exact same rules as function expressions.

---

### Function Expressions with let/const — TDZ instead of undefined

```js
sayHi();  // ❌ ReferenceError: Cannot access 'sayHi' before initialization

const sayHi = function() {
  console.log("Hi!");
};
```

With `const`/`let`, instead of `undefined` you get TDZ.  
So the error is `ReferenceError` instead of `TypeError`.

---

### Function Hoisting Summary

```
SYNTAX                           HOISTED?     CALLABLE BEFORE?
──────────────────────────────────────────────────────────────────────────
function foo() {}                ✅ Fully      ✅ Yes — works perfectly
var foo = function() {}          ⚠️ Partial    ❌ TypeError (var = undefined)
var foo = () => {}               ⚠️ Partial    ❌ TypeError (var = undefined)
const foo = function() {}        ⚠️ TDZ        ❌ ReferenceError
let foo = function() {}          ⚠️ TDZ        ❌ ReferenceError
const foo = () => {}             ⚠️ TDZ        ❌ ReferenceError
```

## Part 3 — Hoisting Priority: Function Declarations Win Over var

When a function declaration and a `var` declaration share the same name, **function declaration wins** during the creation phase.

```js
console.log(foo);  // ƒ foo() { return "function"; } ← function wins

var foo = "I am a variable";

function foo() {
  return "function";
}

console.log(foo);  // "I am a variable" ← reassigned during execution
```

**Step by step:**

```
CREATION PHASE:
  Step 1: var foo → foo = undefined
  Step 2: function foo → foo = function foo() {...}
          ↑ function declaration OVERWRITES the var
  
  End of creation phase: foo = function foo() {...}

EXECUTION PHASE:
  console.log(foo)        → function foo() {...}
  foo = "I am a variable" → foo is now a string
  function foo() {...}    → SKIPPED (already hoisted)
  console.log(foo)        → "I am a variable"
```

> **Priority rule:** `function declarations` > `var` during hoisting.

## Part 4 — Tricky Scenarios (The Real Interview Questions)

---

### Scenario 1 — var hoisted but never assigned

```js
function test() {
  console.log(x);   // ?

  if (false) {
    var x = 10;
  }

  console.log(x);   // ?
}
test();
```

**Answer:** `undefined`, `undefined`.  
`var x` is hoisted to function scope. `if (false)` never runs so assignment never happens. Both logs see `undefined`.

---

### Scenario 2 — Hoisting inside a function (not just global)

```js
function outer() {
  console.log(x);  // ?
  console.log(y);  // ?

  var x = 10;
  let y = 20;
}
outer();
```

**Answer:** `undefined`, then `ReferenceError`.  
Hoisting happens in EVERY execution context — not just global. Every function call creates its own creation phase. `var x` → `undefined`. `let y` → TDZ.

---

### Scenario 3 — Function declaration vs var, same name

```js
console.log(foo);

var foo = "variable";

function foo() { return "function"; }

console.log(foo);
```

**Answer:** `ƒ foo() { return "function"; }`, then `"variable"`.  
Function declaration wins hoisting priority. Then execution reassigns `foo` to the string.

---

### Scenario 4 — Multiple var declarations

```js
var x = 1;
console.log(x);  // ?
var x = 2;
console.log(x);  // ?
var x;
console.log(x);  // ?
```

**Answer:** `1`, `2`, `2`.  
All three `var x` declarations refer to the same variable. The bare `var x` (no assignment) doesn't reset it to `undefined` — it's a re-declaration that does nothing.

---

### Scenario 5 — Function parameters + var hoisting

```js
function foo(a) {
  console.log(a);  // ?
  var a = 10;
  console.log(a);  // ?
}
foo(5);
```

**Answer:** `5`, then `10`.  
Function parameters are part of the function's scope. `var a` inside is a re-declaration of the parameter. The parameter `a = 5` is set before the function body runs. Re-declaration doesn't reset it. Then `a = 10` reassigns it.

---

### Scenario 6 — var expression vs function declaration, called before both

```js
greet();   // ?
sayHi();   // ?

function greet() { console.log("greet"); }
var sayHi = function() { console.log("sayHi"); };
```

**Answer:**  
`greet()` → `"greet"` ✅ (function declaration fully hoisted)  
`sayHi()` → `TypeError: sayHi is not a function` ❌ (`var sayHi` hoisted as `undefined`)

---

### Scenario 7 — let/const expression called before declaration

```js
greet();   // ?
sayHi();   // ?

function greet() { console.log("greet"); }
const sayHi = function() { console.log("sayHi"); };
```

**Answer:**  
`greet()` → `"greet"` ✅  
`sayHi()` → `ReferenceError: Cannot access 'sayHi' before initialization` ❌  
(`const sayHi` in TDZ — different error from `var`)

---

### Scenario 8 — The closure loop (hoisting + scope combined)

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
```

**Answer:** `3, 3, 3`.  
`var i` is hoisted to the enclosing scope — there is ONE shared `i`. Loop finishes (`i = 3`) before timeouts fire. All three closures read the same `i = 3`.

---

### Scenario 9 — Function declaration inside a block (avoid this)

```js
console.log(typeof foo);  // ?

if (true) {
  function foo() { return 1; }
}

console.log(typeof foo);  // ?
```

**Answer:** `"undefined"`, then `"function"` (in most modern browsers, non-strict mode).  
Function declarations inside blocks have inconsistent behavior. **Never do this.**  
Use `const foo = function() {}` instead.

---

### Scenario 10 — Full creation phase mental model test

```js
console.log(a);
console.log(b);
console.log(c);

var a = 1;
let b = 2;
function c() { return 3; }
```

**Answer:**  
`undefined` (var → undefined)  
`ReferenceError: Cannot access 'b' before initialization` (let → TDZ, execution stops here)  
`ƒ c() { return 3; }` would print if `b` didn't throw — function declaration fully hoisted.

## Part 5 — Complete Hoisting Mental Model

```
When JS starts running a script or calls a function:

CREATION PHASE — scans top to bottom:
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  Found function declaration?                             │
│  → Store entire function body in memory                  │
│  → Callable immediately ✅                               │
│                                                          │
│  Found var declaration?                                  │
│  → Store in memory as undefined                          │
│  → Accessible but returns undefined ⚠️                   │
│                                                          │
│  Found let or const declaration?                         │
│  → Store in memory but LOCK in TDZ                       │
│  → NOT accessible — ReferenceError if touched ❌         │
│                                                          │
│  Priority if same name: function > var                   │
│                                                          │
└──────────────────────────────────────────────────────────┘

EXECUTION PHASE — runs line by line:
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  Assignments execute (var gets its value)                │
│  Functions get called                                    │
│  let/const exit TDZ when their line is reached          │
│  Function declaration lines are SKIPPED                  │
│  (already stored during creation phase)                  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## Part 6 — Why This Matters in Real Code

Hoisting causes real bugs — not just quiz questions.

**Bug 1 — Assuming var doesn't exist yet**

```js
var x = 1;

function test() {
  if (x === undefined) {   // dev thinks: x might not be declared yet
    var x = 2;             // but var x is hoisted! x IS undefined here
  }
  console.log(x);          // 2 — confusing and hard to debug
}

test();
```

**Bug 2 — Calling function expression too early**

```js
// somewhere at top of file
var result = calculate(5);  // TypeError if this runs before assignment

// somewhere at bottom
var calculate = function(x) { return x * 2; };
```

**The fix for both:** Use `const`/`let` for variables and function expressions.  
TDZ errors are **loud and immediate** — they tell you exactly what's wrong.  
`var`'s silent `undefined` hides bugs until runtime.

## What SDE2 Interviewers Expect

**1. Explain hoisting using the two-phase mental model**  
Not just *"var is hoisted"*. Say: *"During the creation phase, JS scans all declarations and allocates memory. var is initialized to undefined, let/const go into TDZ, and function declarations are stored fully."*

**2. Know the difference between the two ReferenceErrors**  
*"Cannot access before initialization"* = variable exists but in TDZ (let/const).  
*"Is not defined"* = variable doesn't exist anywhere.  
Interviewers use this to check if you truly understand TDZ vs undeclared.

**3. Know all function types and how they hoist**  
Function declaration → fully hoisted. Function expression with var → undefined. Function expression with const/let → TDZ. This comes up constantly.

**4. Know function declaration beats var in hoisting priority**  
When both share a name, function wins the creation phase. var is overwritten. Then execution can reassign it.

**5. Connect hoisting to scope**  
Hoisting doesn't happen globally for everything — it happens within each scope. `var` hoists to its function scope. `let`/`const` hoist to their block scope. Every function call gets its own creation phase.

**6. Know why TDZ is better than var's undefined**  
TDZ throws immediately, loudly, exactly at the problem line. `var`'s silent `undefined` hides bugs. This is why modern JS uses `const`/`let`.

**7. Never say "let/const are not hoisted"**  
They are. The TDZ proof (inner block shadowing) is a common follow-up question. Be ready to explain it.

---

## The Three Mental Models to Carry Into the Interview

**Hoisting = JS does a dry run before the real run.**  
It reads all declarations and puts them in memory first. Then runs your code. By the time line 1 executes, all declarations already exist in memory.

**var = assigned a placeholder (undefined) right away.**  
It exists from the very start. But its real value only arrives when the assignment line runs.

**let/const = exists but behind a locked door (TDZ).**  
JS knows it's there. But it won't let you touch it until you've formally declared it. The lock opens when the declaration line runs.
