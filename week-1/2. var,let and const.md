# var, let & const. Reassignment. Mutatation
---

## Quick Comparison — The Full Picture

| Property | `var` | `let` | `const` |
|---|---|---|---|
| Scope | Function / global | Block | Block |
| Hoisted | ✅ Yes → `undefined` | ⚠️ Yes → TDZ | ⚠️ Yes → TDZ |
| Re-declarable | ✅ Yes | ❌ No | ❌ No |
| Re-assignable | ✅ Yes | ✅ Yes | ❌ No |
| Mutable (objects) | ✅ Yes | ✅ Yes | ✅ Yes |
| Attaches to `window` | ✅ Yes | ❌ No | ❌ No |
| TDZ | ❌ No | ✅ Yes | ✅ Yes |

## 1. Scope — Where Each Keyword Lives

**The one-line rule:**
- `var` → only a **function** creates a new scope for it. Blocks are invisible to it.
- `let` / `const` → any `{}` creates a new scope.

### var — function scoped (leaks out of blocks)

```js
function test() {
  if (true) {
    var x = 10;     // you think this is inside the if block
  }
  console.log(x);   // 10 ← var LEAKED out of the if block
}

test();
console.log(x);     // ReferenceError ← but did NOT leak out of the function
```

### let and const — block scoped

```js
if (true) {
  let y = 20;
  const z = 30;
}

console.log(y);     // ReferenceError ← block scoped
console.log(z);     // ReferenceError ← block scoped
```

> **Any `{}` counts as a block** — if, for, while, switch, even standalone `{ }`.

## 2. Hoisting — What Actually Happens Before Your Code Runs

### The mental model: Two phases run every time JS executes code

```
┌──────────────────────────────────────────────────────────┐
│                    EXECUTION CONTEXT                     │
│                                                          │
│  ┌───────────────────────────────────────────────────┐   │
│  │  PHASE 1 — Creation Phase  (before code runs)     │   │
│  │                                                   │   │
│  │  JS scans all declarations and allocates memory   │   │
│  │                                                   │   │
│  │  var a = ?    →   allocated, set to undefined     │   │
│  │  let b = ?    →   allocated, put into TDZ ⚠️      │   │
│  │  const c = ?  →   allocated, put into TDZ ⚠️      │   │
│  │                                                   │   │
│  └───────────────────────────────────────────────────┘   │
│                                                          │
│  ┌───────────────────────────────────────────────────┐   │
│  │  PHASE 2 — Execution Phase  (code runs)           │   │
│  │                                                   │   │
│  │  Runs line by line                                │   │
│  │  Assignments happen here                          │   │
│  │  let/const exit TDZ when their line is reached    │   │
│  │                                                   │   │
│  └───────────────────────────────────────────────────┘   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

> **Note:** This two-phase process is part of the **Execution Context** — a bigger concept you'll study on Day 5. Creation phase + execution phase = how every execution context works.

---

### var — hoisted and initialized to `undefined`

```js
// What you write:
console.log(a);   // undefined  ← no error! already exists in memory
var a = 5;
console.log(a);   // 5

// How JS mentally sees this (after creation phase):
var a = undefined;  // ← hoisted to top of scope
console.log(a);     // undefined
a = 5;              // assignment happens at execution time
console.log(a);     // 5
```

---

### let and const — hoisted but sit in TDZ (unusable)

```js
console.log(b);   // ❌ ReferenceError: Cannot access 'b' before initialization
let b = 5;

console.log(c);   // ❌ ReferenceError: Cannot access 'c' before initialization
const c = 10;
```

> **Critical insight:** The error says *"Cannot access before initialization"* — NOT *"b is not defined"*.
>
> This proves `let`/`const` ARE hoisted. JS knows they exist. It just won't let you touch them yet.
> If they weren't hoisted at all, the error would say *"b is not defined"*.

## 3. Temporal Dead Zone (TDZ)

**TDZ = the gap between when a variable is hoisted and when its declaration line is actually executed.**

```
Timeline for: let b = 5;

  Script starts
       │
       ▼
  ┌─────────────────────────────┐
  │  CREATION PHASE             │
  │  b is hoisted → enters TDZ  │ ← b EXISTS but is UNUSABLE
  └─────────────────────────────┘
       │
       │  ...any code here that accesses b → ReferenceError
       │
       ▼
  ┌─────────────────────────────┐
  │  let b = 5;  ← this line   │ ← b EXITS TDZ, gets value 5
  └─────────────────────────────┘
       │
       │  ...b is now accessible, value = 5
       ▼
```

---

### Memory state during TDZ vs after

```
CREATION PHASE              AFTER DECLARATION LINE
──────────────────          ──────────────────────
var a  → undefined          var a  → undefined
let b  → [TDZ] ⚠️           let b  → 5 ✓
const c → [TDZ] ⚠️          const c → 10 ✓
```

---

### The hardest TDZ question — inner block shadowing

```js
let x = 1;

{
  console.log(x);   // ❌ ReferenceError — most people expect 1 here!
  let x = 2;
}
```

**Why?** The inner `let x` is hoisted to the top of its block. This creates a TDZ for `x` from the very start of that block. It **shadows** the outer `x = 1`. So `console.log(x)` hits the inner TDZ variable — not the outer one.

> If the inner `let x = 2` wasn't there, `console.log(x)` would print `1` (outer scope). The inner declaration is what creates the shadow.

## 4. Re-declaration and Re-assignment

```js
// var — silently re-declarable (dangerous!)
var a = 1;
var a = 2;      // ✅ no error, overwrites silently
console.log(a); // 2

// let — cannot re-declare in same scope
let b = 1;
let b = 2;      // ❌ SyntaxError: Identifier 'b' already declared

// const — cannot re-declare or re-assign
const c = 1;
const c = 2;    // ❌ SyntaxError
c = 99;         // ❌ TypeError: Assignment to constant variable

// const with objects — reference is locked, contents are not
const obj = { x: 1 };
obj.x = 99;        // ✅ mutation is fine
obj.newProp = "hi"; // ✅ adding props is fine
obj = { x: 1 };    // ❌ TypeError — reassigning the variable
```

---

## 5. const — Full Picture

```js
// Must be initialized at declaration — no "declare now, assign later"
const x;      // ❌ SyntaxError: Missing initializer in const declaration
const x = 5;  // ✅

// Primitives — fully locked
const n = 10;
n = 20;       // ❌ TypeError

// Arrays — same as objects
const arr = [1, 2, 3];
arr.push(4);  // ✅ mutation ok
arr[0] = 99;  // ✅ mutation ok
arr = [];     // ❌ TypeError
```

### Want true immutability? Use Object.freeze()

```js
const frozen = Object.freeze({ a: 1, nested: { b: 2 } });

frozen.a = 99;         // silently fails (throws in strict mode)
frozen.nested.b = 99;  // ✅ — Object.freeze() is SHALLOW
                        //    nested objects are NOT frozen!
```

> For **deep immutability** you need a recursive freeze or a library like Immer.

## 6. var and the Global Object

```js
// In browser, at global scope:
var name = "Alice";
console.log(window.name);  // "Alice" ← var attaches to window!

let age = 30;
console.log(window.age);   // undefined ← let does NOT attach to window

const city = "Jaipur";
console.log(window.city);  // undefined ← const does NOT either
```

**Why this is dangerous:**
```js
var top = 10;
// window.top is a built-in browser property (reference to topmost window)
// you just silently overrode it!
```

> `let`/`const` never touch `window`. This alone is reason enough to never use `var`.

## 7. The Classic for Loop Bug — Most Asked Interview Question

### With var — broken

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}

// Output: 3, 3, 3  ← NOT what you'd expect
```

### With let — fixed

```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}

// Output: 0, 1, 2  ✓
```

### Root cause — visualized

```
WITH var:
┌─────────────────────────────────────────┐
│  ONE shared 'i' in memory               │
│                                         │
│  Iteration 0: i=0 → callback queued     │
│  Iteration 1: i=1 → callback queued     │
│  Iteration 2: i=2 → callback queued     │
│  Loop ends:   i=3 ← all 3 callbacks     │
│               fire now, read i=3        │
└─────────────────────────────────────────┘
Output: 3, 3, 3

WITH let:
┌─────────────────────────────────────────┐
│  NEW 'i' binding created each iteration │
│                                         │
│  Iteration 0: i₀=0 → callback captures i₀
│  Iteration 1: i₁=1 → callback captures i₁
│  Iteration 2: i₂=2 → callback captures i₂
│  Each callback has its OWN i            │
└─────────────────────────────────────────┘
Output: 0, 1, 2
```

### The old fix before let existed — IIFE

```js
for (var i = 0; i < 3; i++) {
  (function(j) {
    setTimeout(() => console.log(j), 1000);
  })(i);  // immediately invoked — creates its own scope with its own j
}

// Output: 0, 1, 2
```

> Knowing the IIFE fix proves you understand the **root cause** — not just that `let` fixes it.

## 8. Function Declaration Hoisting — Different Rules

```js
// Function DECLARATIONS — fully hoisted (name + entire body)
greet();                          // ✅ "Hello!" — works before the declaration
function greet() {
  console.log("Hello!");
}

// Function EXPRESSIONS — NOT fully hoisted
sayHi();                          // ❌ TypeError: sayHi is not a function
var sayHi = function() {
  console.log("Hi!");
};
// var sayHi IS hoisted as undefined
// calling undefined() → TypeError
```

```
CREATION PHASE — what's in memory:

greet  → [entire function body] ✓   ← declaration: fully hoisted
sayHi  → undefined                  ← expression: only var is hoisted
```

### Avoid function declarations inside blocks

```js
// Behavior is inconsistent across environments — avoid this pattern
if (true) {
  function foo() { return 1; }
} else {
  function foo() { return 2; }
}

// Use this instead:
const foo = true ? () => 1 : () => 2;
```

## 9. When to Use Which

| Use | When |
|---|---|
| `const` | **Default — always start here.** Most variables don't need reassignment. |
| `let` | When you know the value needs to change — loop counters, accumulators. |
| `var` | **Never.** No reason to use it in modern JS. |

## 10. Output Questions — Work Through Before Reading Answers

**Q1 — Basic var hoisting**
```js
console.log(a);
var a = 5;
console.log(a);
```
<details>
<summary>Answer</summary>

`undefined`, then `5`.
`var a` is hoisted and set to `undefined` in creation phase. The assignment `a = 5` happens at execution time.
</details>

---

**Q2 — let in TDZ**
```js
console.log(b);
let b = 5;
```
<details>
<summary>Answer</summary>

`ReferenceError: Cannot access 'b' before initialization`
`let b` is hoisted but sits in TDZ. Accessing it before its line = error.
</details>

---

**Q3 — var scope leak from loop**
```js
for (var i = 0; i < 3; i++) {}
console.log(i);
```
<details>
<summary>Answer</summary>

`3`
`var` leaks out of the for loop block. `i` exists in the enclosing function/global scope with value `3`.
</details>

---

**Q4 — let scope**
```js
for (let j = 0; j < 3; j++) {}
console.log(j);
```
<details>
<summary>Answer</summary>

`ReferenceError: j is not defined`
`let` is block-scoped. `j` does not exist outside the for loop.
</details>

---

**Q5 — The closure loop (most asked)**
```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
```
<details>
<summary>Answer</summary>

`3, 3, 3`
One shared `var i`. Loop finishes (`i = 3`) before any setTimeout fires. All 3 callbacks close over the same `i = 3`.
</details>

---

**Q6 — Same loop with let**
```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
```
<details>
<summary>Answer</summary>

`0, 1, 2`
`let` creates a new `i` binding per iteration. Each callback captures its own separate `i`.
</details>

---

**Q7 — Inner block TDZ shadow (hardest)**
```js
let x = 1;
{
  console.log(x);
  let x = 2;
}
```
<details>
<summary>Answer</summary>

`ReferenceError`
The inner `let x` is hoisted to the top of its block, creating TDZ from the block start. It shadows the outer `x = 1`. `console.log(x)` hits the TDZ inner variable — most people expect `1`.
</details>

---

**Q8 — Re-declaration**
```js
var a = 1;
var a = 2;
console.log(a);

let b = 1;
let b = 2;
```
<details>
<summary>Answer</summary>

`2` is logged (var re-declaration succeeds silently).
Then `SyntaxError` on `let b = 2` — caught at parse time, before any code runs.
</details>

---

**Q9 — const + mutation**
```js
const obj = { x: 1 };
obj.x = 99;
console.log(obj.x);
obj = { x: 2 };
```
<details>
<summary>Answer</summary>

`99` is logged.
Then `TypeError: Assignment to constant variable` — mutation is fine, reassignment is not.
</details>

---

**Q10 — Function declaration vs expression hoisting**
```js
greet();
sayHi();
function greet() { console.log("greet"); }
var sayHi = function() { console.log("sayHi"); };
```
<details>
<summary>Answer</summary>

`greet()` → `"greet"` — function declarations are fully hoisted.
`sayHi()` → `TypeError: sayHi is not a function` — `var sayHi` hoisted as `undefined`, not the function body.
</details>

## What Interviewers Expect

**1. Explain hoisting with the creation phase mental model**
Not just *"var is hoisted"*. Say: JS allocates memory before execution. `var` = `undefined`. `let`/`const` = TDZ. This two-phase process is part of the Execution Context.

**2. Explain TDZ precisely — never say "let isn't hoisted"**
`let`/`const` ARE hoisted. The error *"cannot access before initialization"* proves it. *"Is not defined"* would mean it wasn't hoisted at all. These are two different errors.

**3. Nail the for loop bug AND explain the root cause**
`var` = one shared binding across iterations. `let` = new binding per iteration. Bonus: know the IIFE fix — it shows you understand the mechanism.

**4. Know `const` ≠ immutable**
`const` prevents reassignment of the variable, not mutation of what it points to. `Object.freeze()` for shallow immutability. Nested objects still mutable.

**5. Know `var` pollutes `window`**
`var` at global scope adds a property to `window`. `let`/`const` do not. Can silently override browser built-ins.

**6. Preference order when asked**
`const` by default → `let` when reassignment is needed → `var` never in modern code.

**7. Connect TDZ to execution context**
Be able to say: *"The creation phase of the execution context is where hoisting happens — var gets initialized to undefined, let and const are placed in the TDZ until their declaration line is reached during the execution phase."*
