## Part 1 — Temporal Dead Zone (TDZ)

### What is TDZ?

`let` and `const` are hoisted — but instead of being initialized to `undefined` like `var`, they are placed in a **locked state** called the Temporal Dead Zone.

TDZ = the period between when a variable is **hoisted** and when its **declaration line is executed**.  
Any access during this period throws a `ReferenceError`.

```
Script starts
     │
     ▼
┌─────────────────────────────────┐
│  CREATION PHASE                 │
│  let b → hoisted → enters TDZ  │ ← b EXISTS but LOCKED ⚠️
└─────────────────────────────────┘
     │
     │  ← any access to b here = ReferenceError ❌
     │
     ▼
┌─────────────────────────────────┐
│  let b = 5  ← this line runs   │ ← b EXITS TDZ, gets value 5 ✅
└─────────────────────────────────┘
     │
     │  ← b is now accessible ✅
     ▼
```

```js
console.log(b); // ❌ ReferenceError: Cannot access 'b' before initialization
let b = 5;
console.log(b); // ✅ 5
```

---

### Why TDZ Exists

Before `let`/`const`, only `var` existed. `var` gave `undefined` silently — hiding bugs.  
TDZ was introduced to make bugs **loud and immediate**.

```js
console.log(x); // undefined — silent bug, hard to trace (var)
var x = 5;

console.log(y); // ReferenceError — loud, immediate, tells you exactly what's wrong (let)
let y = 5;
```

---

### Proof That let/const ARE Hoisted

People often say *"let/const are not hoisted"*. That's wrong. Here's the proof:

```js
let x = 1;
{
  console.log(x); // ❌ ReferenceError — NOT 1!
  let x = 2;
}
```

If the inner `let x` was NOT hoisted, `console.log(x)` would look outward and find `x = 1` → print `1`.  
But the inner `let x` IS hoisted to the top of its block → enters TDZ → shadows the outer `x`.  
So `console.log(x)` hits the TDZ variable → ReferenceError.

**The error message is the proof:**
```
"Cannot access 'x' before initialization"
→ JS knows x EXISTS (it was hoisted), just won't allow access yet

"x is not defined"
→ x doesn't exist anywhere — completely different situation
```

---

### typeof + TDZ — The One Exception

`typeof` is normally 100% safe — never throws, even for undeclared variables:

```js
console.log(typeof undeclaredVar); // "undefined" — no error
```

But in TDZ, `typeof` DOES throw:

```js
console.log(typeof x); // ❌ ReferenceError: Cannot access 'x' before initialization
let x = 5;
```

> This is the **only exception** to "typeof never throws". TDZ breaks that rule.

---

### Function Default Parameters + TDZ

```js
function foo(a = b, b = 2) {
  return a + b;
}
foo(); // ❌ ReferenceError
```

Parameters are evaluated **left to right**. When `a = b` evaluates, `b` is still in TDZ.

```js
// Fix — swap the order
function foo(a = 2, b = a) {
  return a + b;
}
foo(); // ✅ 4
```

---

### TDZ Summary Table

```
                    var             let / const
─────────────────────────────────────────────────────
Hoisted?           Yes             Yes
Initialized to     undefined       TDZ (locked)
Accessible before? Yes (undefined) No (ReferenceError)
typeof safe?       Yes             No — throws in TDZ
```

## Part 2 — Type Coercion

### What is Type Coercion?

JavaScript is **loosely typed** — you never declare what type a variable is.  
When you do an operation between two different types, JS **automatically converts** one type to another.  
This automatic conversion is called **type coercion**.

```js
"5" + 1   // "51"  ← number converted to string
"5" - 1   // 4     ← string converted to number
true + 1  // 2     ← boolean converted to number
```

---

### Implicit vs Explicit Coercion

**Implicit** — JS does it automatically:
```js
"5" * 2    // 10   ← JS converted "5" to number
if (0) {}  // 0 coerced to false
```

**Explicit** — you do it yourself:
```js
Number("5")   // 5
String(42)    // "42"
Boolean(0)    // false
```

> Interviews mostly test **implicit coercion** — that's where the tricky output questions live.

---

### Converting to Number — The Rules

```js
Number("42")       // 42      ← valid number string
Number("3.14")     // 3.14
Number("")         // 0       ← empty string → 0
Number("  ")       // 0       ← whitespace only → 0
Number("abc")      // NaN     ← invalid
Number("12abc")    // NaN     ← can't fully parse
Number(true)       // 1
Number(false)      // 0
Number(null)       // 0       ← null → 0
Number(undefined)  // NaN     ← undefined → NaN
Number([])         // 0       ← empty array → 0
Number([3])        // 3       ← single item array → that number
Number([1,2])      // NaN     ← multiple items → NaN
Number({})         // NaN     ← object → NaN
```

> **Must memorize:** `null → 0`, `undefined → NaN`, `"" → 0`, `true → 1`, `false → 0`

---

### Converting to String — The Rules

```js
String(42)         // "42"
String(3.14)       // "3.14"
String(true)       // "true"
String(false)      // "false"
String(null)       // "null"
String(undefined)  // "undefined"
String([1,2,3])    // "1,2,3"
String({})         // "[object Object]"
```

---

### Converting to Boolean — Truthy and Falsy

Every value in JS is either **truthy** or **falsy** when converted to boolean.

#### The 8 Falsy Values — Memorize All of These

```js
Boolean(false)     // false
Boolean(0)         // false
Boolean(-0)        // false
Boolean(0n)        // false  ← BigInt zero
Boolean("")        // false  ← empty string only
Boolean(null)      // false
Boolean(undefined) // false
Boolean(NaN)       // false
```

> **Memory trick:** `false, 0, -0, 0n, "", null, undefined, NaN` — that's the complete list.  
> **Everything else is truthy.**

#### The Truthy Traps — These Catch Everyone

```js
Boolean("0")       // true  ← string "0" is NOT falsy (not empty string)
Boolean("false")   // true  ← non-empty string, always truthy
Boolean(" ")       // true  ← space is NOT empty string
Boolean([])        // true  ← empty array is TRUTHY ← very common trap
Boolean({})        // true  ← empty object is TRUTHY ← very common trap
Boolean(-1)        // true  ← any non-zero number is truthy
Boolean(Infinity)  // true
Boolean([0])       // true  ← array with falsy value inside is still truthy
Boolean(function(){}) // true
```

> **The two biggest traps:** `[]` and `{}` are both **truthy** even though they're "empty".

## Part 3 — == vs ===

### === (Strict Equality) — No Coercion, Ever

Checks **value AND type**. If types differ → immediately `false`. No conversion happens.

```js
5 === 5           // true  ← same type, same value
5 === "5"         // false ← different types → false immediately
null === null     // true
null === undefined // false ← different types
false === 0       // false ← different types
```

---

### == (Loose Equality) — Coerces Types First

Checks value but **converts types** if they differ. Follows a specific set of rules.

---

### The == Rules — In Order

#### Rule 1 — Same type? Compare directly (same as ===)

```js
5 == 5       // true
"hi" == "hi" // true
```

---

#### Rule 2 — null and undefined: only equal to each other

```js
null == undefined  // true  ← special pair, hardcoded rule
null == 0          // false ← null doesn't coerce to 0 in ==
null == ""         // false
null == false      // false
undefined == 0     // false
undefined == false // false
```

> `null` and `undefined` only `==` each other. Nothing else. No exceptions.

---

#### Rule 3 — NaN: never equal to anything, including itself

```js
NaN == NaN   // false ← always false, no exceptions
NaN === NaN  // false
```

> To check for NaN: use `Number.isNaN(x)` or `x !== x` (only NaN fails self-equality check).

---

#### Rule 4 — Number vs String → convert string to number, then compare

```js
5 == "5"    // true  ← "5" → 5 → 5 == 5
0 == ""     // true  ← "" → 0 → 0 == 0
0 == "0"    // true  ← "0" → 0 → 0 == 0
1 == "1"    // true
2 == "2"    // true
```

---

#### Rule 5 — Boolean vs anything → convert boolean to number FIRST, then reapply rules

```js
true == 1      // true  ← true → 1 → 1 == 1
false == 0     // true  ← false → 0 → 0 == 0
true == "1"    // true  ← true → 1, "1" → 1 → 1 == 1
true == "2"    // false ← true → 1, "2" → 2 → 1 == 2
false == ""    // true  ← false → 0, "" → 0 → 0 == 0
false == "0"   // true  ← false → 0, "0" → 0 → 0 == 0
false == null  // false ← false → 0, null stays (Rule 2 takes over)
false == undefined // false
```

> **Key:** boolean always converts to number first. THEN the comparison restarts from the top.

---

#### Rule 6 — Object vs primitive → convert object to primitive, then reapply rules

JS calls `.valueOf()` then `.toString()` on the object.

```js
[] == 0        // true  ← [] → "" → 0 → 0 == 0
[] == ""       // true  ← [] → "" → "" == ""
[] == false    // true  ← false → 0, [] → "" → 0 → 0 == 0
[1] == 1       // true  ← [1] → "1" → 1 → 1 == 1
[1] == "1"     // true  ← [1] → "1" → "1" == "1"
[[]] == 0      // true  ← [[]] → "" → 0 → 0 == 0
```

---

### The Most Famous == Trap — `[] == ![]`

```js
[] == ![]  // true — the most surprising result in JS
```

Step by step:
```
Step 1: Evaluate ![] first
        [] is truthy → ![] = false

Step 2: [] == false
        false is boolean → convert to number → 0
        Now: [] == 0

Step 3: [] is object, 0 is number
        Convert [] to primitive → [] → "" (empty string)
        Now: "" == 0

Step 4: "" is string, 0 is number
        Convert "" to number → 0
        Now: 0 == 0 → true ✅
```

> If you can explain this step by step in an interview, you will genuinely impress the interviewer.

---

### == Decision Flowchart

```
a == b  (types are different)
│
├─ Is one null and the other undefined?
│   └─ YES → true (special pair)
│
├─ Is either value NaN?
│   └─ YES → false (always)
│
├─ Is one a boolean?
│   └─ YES → convert boolean to number → restart comparison
│
├─ Is one a string and other a number?
│   └─ YES → convert string to number → compare
│
├─ Is one an object and other a primitive?
│   └─ YES → convert object to primitive → restart comparison
│
└─ Otherwise → false
```

---

### == vs === Summary

```
===  →  No coercion. Types differ = false. Always predictable.
==   →  Coerces types. Follows 6 rules. Often surprising.

Use ===  →  always, by default
Use ==   →  only for null/undefined check:

if (value == null) {
  // catches BOTH null AND undefined
  // cleaner than: value === null || value === undefined
}
```

## Part 4 — Coercion in Operators

### + operator — prefers string (if any operand is a string)

```js
1 + "2"         // "12"  ← number → string → concatenate
"3" + 4 + 5     // "345" ← left to right: "3"+4="34", "34"+5="345"
3 + 4 + "5"     // "75"  ← left to right: 3+4=7, 7+"5"="75"
true + true     // 2     ← 1 + 1
true + true + "1" // "21" ← 1+1=2, then 2+"1"="21" (STRING, not number 21)
1 + null        // 1     ← null → 0
1 + undefined   // NaN   ← undefined → NaN
```

> **The `+` trap:** `true + true + "1"` = `"21"` (string) not `21` (number).  
> Arithmetic happens first (2), then string concatenation kicks in ("21").

---

### -, *, / operators — always convert to number

```js
"5" - 2       // 3    ← "5" → 5
"5" * "2"     // 10   ← both → number
null - 1      // -1   ← null → 0
undefined - 1 // NaN  ← undefined → NaN
true - false  // 1    ← 1 - 0
"" - 1        // -1   ← "" → 0
```

---

### Unary + — quick way to convert to number

```js
+"42"       // 42
+""         // 0
+true       // 1
+false      // 0
+null       // 0
+undefined  // NaN
+[]         // 0
+{}         // NaN
+"abc"      // NaN
```

## Part 5 — Logical Operators + Coercion

`&&` and `||` don't return `true`/`false` — they return one of the **actual operand values**.

### || (OR) — returns first TRUTHY value, or last value if all falsy

```js
"hello" || "world"    // "hello" ← first truthy, stops here
0 || "world"          // "world" ← 0 is falsy, moves on
null || undefined     // undefined ← both falsy, returns last
false || 0 || "yes"   // "yes"   ← first truthy found
0 || ""               // ""      ← both falsy, returns last
```

### && (AND) — returns first FALSY value, or last value if all truthy

```js
"hello" && "world"    // "world" ← both truthy, returns last
0 && "world"          // 0       ← first falsy, stops here
"yes" && 42           // 42      ← both truthy, returns last
null && "anything"    // null    ← first falsy, stops here
1 && 2 && 3           // 3       ← all truthy, returns last
```

---

### ?? (Nullish Coalescing) vs || (OR) — Critical Difference

```js
// || falls back for ALL falsy values (0, "", false, null, undefined, NaN)
const a = 0 || "default"    // "default" ← 0 is falsy
const b = "" || "default"   // "default" ← "" is falsy

// ?? only falls back for null and undefined — nothing else
const c = 0 ?? "default"    // 0   ← 0 is not null/undefined, keeps it
const d = "" ?? "default"   // ""  ← kept
const e = null ?? "default" // "default" ← null triggers fallback
const f = undefined ?? "default" // "default" ← undefined triggers fallback
```

> **Use `??` when `0` or `""` are valid values you want to keep.**  
> **Use `||` when you want to fall back on any falsy value.**

## Output Questions — Work Through Before Reading Answers

**Q1 — == with boolean and number**
```js
console.log(0 == false);
console.log(0 === false);
```
<details>
<summary>Answer</summary>

`true` — `false` → `0`, then `0 == 0` → true.  
`false` — `===` never coerces. Different types (number vs boolean) → false immediately.
</details>

---

**Q2 — == with empty string and boolean**
```js
console.log("" == false);
console.log("" === false);
```
<details>
<summary>Answer</summary>

`true` — `false` → `0`, then `""` → `0`, then `0 == 0` → true.  
`false` — `===` no coercion. Different types → false.
</details>

---

**Q3 — null and undefined**
```js
console.log(null == undefined);
console.log(null === undefined);
```
<details>
<summary>Answer</summary>

`true` — special hardcoded rule: `null` and `undefined` are only `==` to each other.  
`false` — `===` requires same type. `null` and `undefined` are different types.
</details>

---

**Q4 — array and boolean**
```js
console.log([] == false);
console.log([] === false);
```
<details>
<summary>Answer</summary>

`true` — `false` → `0`, `[]` → `""` → `0`, `0 == 0` → true.  
`false` — different types, `===` returns false immediately.
</details>

---

**Q5 — NaN**
```js
console.log(NaN == NaN);
console.log(NaN === NaN);
```
<details>
<summary>Answer</summary>

Both `false`. NaN is never equal to anything, including itself. Use `Number.isNaN(x)` to check for NaN.
</details>

---

**Q6 — typeof in TDZ**
```js
console.log(typeof x);
let x = 5;
```
<details>
<summary>Answer</summary>

`ReferenceError: Cannot access 'x' before initialization`.  
`let x` is in TDZ. `typeof` normally never throws — but TDZ is the one exception.
</details>

---

**Q7 — + operator with strings (left to right)**
```js
console.log(1 + "2" + 3);
console.log(1 + 2 + "3");
```
<details>
<summary>Answer</summary>

`"123"` — `1 + "2"` = `"12"` (string), then `"12" + 3` = `"123"`.  
`"33"` — `1 + 2` = `3` (number), then `3 + "3"` = `"33"`.  
Left-to-right evaluation. Type of result depends on when the string appears.
</details>

---

**Q8 — The type trap**
```js
console.log(true + true + "1");
```
<details>
<summary>Answer</summary>

`"21"` — a STRING, not the number 21.  
`true + true` = `1 + 1` = `2` (number). Then `2 + "1"` = `"21"` (string).  
`typeof (true + true + "1")` = `"string"`.
</details>

---

**Q9 — null and undefined in arithmetic**
```js
console.log(null + 1);
console.log(undefined + 1);
```
<details>
<summary>Answer</summary>

`1` — `null` → `0`, `0 + 1` = `1`.  
`NaN` — `undefined` → `NaN`, `NaN + 1` = `NaN`.
</details>

---

**Q10 — The wildest one**
```js
console.log([] == ![]);
```
<details>
<summary>Answer</summary>

`true`.  
Step 1: `![]` → `false` (arrays are truthy, negating = false).  
Step 2: `[] == false` → `false` → `0`.  
Step 3: `[] == 0` → `[]` → `""`.  
Step 4: `"" == 0` → `""` → `0` → `0 == 0` → `true`.
</details>

**Q11 — truthy/falsy traps**
```js
console.log(Boolean("0"));
console.log(Boolean([]));
console.log(Boolean({}));
console.log(Boolean("false"));
console.log(Boolean(" "));
```
<details>
<summary>Answer</summary>

All `true`.  
`"0"` → non-empty string → truthy.  
`[]` → empty array → truthy (arrays are always truthy).  
`{}` → empty object → truthy (objects are always truthy).  
`"false"` → non-empty string → truthy.  
`" "` → space is not empty string → truthy.
</details>

**Q12 — || and && return values**
```js
console.log(0 || "hello");
console.log(1 && "hello");
console.log(null || undefined);
console.log("a" && "b" && "c");
```
<details>
<summary>Answer</summary>

`"hello"` — `0` is falsy, `||` moves to next, returns `"hello"`.  
`"hello"` — `1` is truthy, `&&` moves to next, both truthy, returns last: `"hello"`.  
`undefined` — both falsy, `||` returns last value.  
`"c"` — all truthy, `&&` returns last value.
</details>

**Q13 — ?? vs ||**
```js
console.log(0 || "default");
console.log(0 ?? "default");
console.log("" || "default");
console.log("" ?? "default");
console.log(null ?? "default");
```
<details>
<summary>Answer</summary>

`"default"` — `||`: `0` is falsy, falls back.  
`0` — `??`: `0` is not `null`/`undefined`, keeps it.  
`"default"` — `||`: `""` is falsy, falls back.  
`""` — `??`: `""` is not `null`/`undefined`, keeps it.  
`"default"` — `??`: `null` triggers fallback.
</details>

## What SDE2 Interviewers Expect

**1. Know the 8 falsy values without hesitation**  
`false, 0, -0, 0n, "", null, undefined, NaN`. Everything else is truthy — especially `[]`, `{}`, `"0"`, `"false"`.

**2. Know the == rules in order**  
null/undefined special pair → NaN always false → boolean converts to number → string vs number converts string → object vs primitive converts object. Explain step by step.

**3. Always use === unless you have a specific reason**  
The one valid use of `==`: `value == null` catches both `null` and `undefined` cleanly.

**4. Explain [] == ![] step by step**  
This is the ultimate coercion question. Four steps. If you can walk through it, you know coercion cold.

**5. Know TDZ breaks typeof**  
typeof normally never throws. TDZ is the one exception. Very common follow-up.

**6. Know ?? vs || difference**  
`||` falls back on any falsy. `??` only falls back on `null`/`undefined`. Real-world usage: prefer `??` for default values when `0` or `""` are valid.

**7. Know + operator left-to-right evaluation**  
`1 + 2 + "3"` = `"33"`. `"1" + 2 + 3` = `"123"`. The position of the string determines when concatenation kicks in.

**8. Know && and || return actual values, not booleans**  
`||` returns first truthy or last value. `&&` returns first falsy or last value. This is how default value patterns work.

## The Mental Models to Carry Into the Interview

**Coercion = JS trying to make the operation work.**  
It has a rulebook. Learn the rulebook and coercion becomes completely predictable.

**=== = strict bouncer. Checks ID (type) before anything.**  
Wrong type? You're not getting in. No exceptions.

**== = lenient bouncer. Will convert your ID first if needed.**  
Follows a specific set of rules to convert. Know the rules, know the result.

**Falsy = the 8 outsiders. Everything else is truthy.**  
The two biggest surprises: `[]` and `{}` are truthy. `"0"` is truthy. Memorize the 8.

**TDZ = a locked door with a sign.**  
JS put the variable behind the door during creation phase. The sign says "not ready yet". The door unlocks when the declaration line runs.
