# Garbage Collection + Memory Leaks


## The Big Picture First

When JS runs your code, it writes things to memory — variables, objects, functions, arrays.  
Memory is limited. If things keep getting written and never cleaned up, your app slows down and eventually crashes.

**Garbage Collection = JS's automatic memory cleaner.**

JS has a background process that periodically looks at everything in memory and asks:  
*"Is this still needed by the program?"*  
If not — it erases it automatically. You never call it manually.

---

## Stack vs Heap — Where the Problem Lives

You already know this from Primitives vs Reference:

```
STACK MEMORY                        HEAP MEMORY
────────────────────────            ────────────────────────
Stores primitives                   Stores objects, arrays,
and variable references             functions — reference types

Cleans itself automatically         Needs garbage collector
When a function returns,            Objects stay until GC
its EC is popped and all            determines they're unreachable
its variables are gone
```

**Stack = no problem. Heap = where GC works.**

```js
function greet() {
  let name = "Alice";  // stack memory
}                      // function returns → EC popped → name gone ✅

let user = { age: 25 }; // object on HEAP
                         // stays until GC decides it's unreachable
```

---

## Reachability — The Core Concept

The entire garbage collection system is built on one idea:

> **If a value can be reached from anywhere in the running program — keep it.**  
> **If it cannot be reached from anywhere — it's garbage, free it.**

---

## What Are Roots?

Roots are the **starting points** JS uses to check what's reachable.  
Think of roots as the "front doors" of your program — always accessible, always alive.

```
ROOTS (always reachable, never collected):
─────────────────────────────────────────
1. Global variables
   → anything at the top level of your script
   → window.x, any var/let/const outside all functions

2. Currently executing code
   → variables in functions currently on the call stack
   → if a function is running right now, its variables are roots

Everything else must be reachable FROM a root to stay alive.
```

## Mark and Sweep — How GC Actually Works

Modern JS engines use the **Mark and Sweep** algorithm.

### Step 1 — Mark

Start from every root. Follow every reference. Mark everything reachable as "alive".

```
ROOTS
  │
  ├──▶ Object A  ← MARKED ✅
  │         │
  │         └──▶ Object B  ← MARKED ✅
  │
  └──▶ Object C  ← MARKED ✅

Object D  ← no path from any root → NOT MARKED ❌
Object E  ← no path from any root → NOT MARKED ❌
```

### Step 2 — Sweep

Go through all heap memory. Anything not marked = garbage → erase it.

```
Object A → marked → KEEP ✅
Object B → marked → KEEP ✅
Object C → marked → KEEP ✅
Object D → not marked → FREE 🗑️
Object E → not marked → FREE 🗑️
```

> GC runs this cycle periodically in the background. You never trigger it manually.

---

## Memory Lifecycle in JS

Every piece of memory goes through 3 stages:

```
┌──────────────────────────────────────────────────────────┐
│                    MEMORY LIFECYCLE                      │
│                                                          │
│  1. ALLOCATE                                             │
│     JS allocates memory when you create things           │
│     let x = 5           → stack allocated               │
│     let obj = {}        → heap allocated                │
│     function foo() {}   → heap allocated                │
│                                                          │
│  2. USE                                                  │
│     Read and write to the memory                         │
│     obj.name = "Alice"                                   │
│     console.log(obj.name)                                │
│                                                          │
│  3. RELEASE                                              │
│     Memory no longer needed                              │
│     GC frees it when unreachable                         │
│     Happens automatically — you don't call it            │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

---

## Memory Leaks — What They Are

> **A memory leak = memory you no longer need, but JS can't free because something still references it.**

JS isn't broken. GC isn't broken.  
You just accidentally left a reference somewhere — so JS thinks the object is still needed.

Over time, leaked memory builds up → app gets slower → eventually crashes.

```
Normal situation:
  You're done with object → no references left → GC frees it ✅

Memory leak:
  You're done with object → but a reference still exists somewhere
  → JS thinks it's still needed → GC never frees it ❌
  → memory keeps growing
```

---

## The 4 Memory Leak Patterns

### Leak 1 — Forgotten Timer (setInterval / setTimeout)

```js
let bigData = { /* large object */ };

setInterval(function() {
  console.log(bigData); // bigData referenced inside callback
}, 1000);
```

What's happening:

```
ROOT
 │
 └──▶ setInterval (always running)
           │
           └──▶ callback function
                      │
                      └──▶ bigData ──▶ { large object in heap }
                           ↑
                    ALWAYS REACHABLE — never freed ❌
```

The interval runs every second. Its callback references `bigData`. As long as the interval runs, `bigData` is reachable. GC can never clean it. It stays in memory forever.

**The fix — clear the interval when done:**

```js
let bigData = { /* large object */ };

let intervalId = setInterval(function() {
  console.log(bigData);
}, 1000);

// When you're done:
clearInterval(intervalId); // stops interval → callback released
                            // bigData becomes unreachable → GC frees it ✅
```

---

### Leak 2 — Event Listener Not Removed

```js
function setup() {
  let bigData = new Array(10000).fill("data"); // large object

  document.getElementById("btn").addEventListener("click", function() {
    console.log("clicked");
    // This is a closure
    // It captures the entire scope of setup()
    // Including bigData — even though it never uses it!
  });
}

setup();
```

What's happening:

```
ROOT
 │
 └──▶ button (in DOM — always reachable)
           │
           └──▶ click listener (closure)
                      │
                      └──▶ bigData ──▶ { 10,000 items in heap }
                           ↑
                    ALWAYS REACHABLE via DOM → button → listener ❌
```

The button is in the DOM. DOM is always reachable. Button holds the listener. Listener is a closure that captured `bigData`. So `bigData` is stuck in memory forever.

**The fix — remove the listener when done:**

```js
function handleClick() {
  console.log("clicked");
  // separate named function — no closure over bigData
}

document.getElementById("btn").addEventListener("click", handleClick);

// When done with the button:
document.getElementById("btn").removeEventListener("click", handleClick);
// listener released → closure gone → bigData can be GC'd ✅
```

---

### Leak 3 — Accidental Global Variables

```js
function foo() {
  leakedVar = "oops"; // no let/const/var — goes to window!
}

foo();
// window.leakedVar = "oops" now exists
// window is a root — always reachable
// leakedVar stays in memory for the entire page lifetime ❌
```

What's happening:

```
ROOT
 │
 └──▶ window (always a root)
           │
           └──▶ window.leakedVar = "oops"
                ↑
           ALWAYS REACHABLE — never freed until page closes ❌
```

**The fix — always declare variables properly:**

```js
"use strict"; // throws ReferenceError for undeclared variables

function foo() {
  let leakedVar = "safe"; // properly declared
}                          // EC popped → leakedVar gone ✅
```

---

### Leak 4 — Detached DOM Nodes

```js
let savedDiv; // global variable

function removeDiv() {
  let div = document.getElementById("myDiv");
  document.body.removeChild(div); // removed from the PAGE ✅
  savedDiv = div;                  // but saved in global variable ❌
}

removeDiv();
```

What's happening:

```
The page (DOM):               Memory (heap):
────────────────              ──────────────────────────
body                          savedDiv ──▶ { div element }
 └── (myDiv is GONE) ✅                    ↑
                              Still in memory — LEAK ❌
                              "Ghost" — invisible on page,
                              alive in memory
```

The div is removed from the visible page. But `savedDiv` (a global variable = root) still holds a reference to it. GC can't free it. This is called a **detached DOM node**.

**The fix — null out the reference when done:**

```js
function removeDiv() {
  let div = document.getElementById("myDiv");
  document.body.removeChild(div);
  // don't store the reference
  // if you must temporarily: set to null when done
  savedDiv = null; // reference released → GC can free the node ✅
}
```

## WeakMap and WeakRef — References That Don't Prevent GC

### Strong vs Weak References

Every normal variable holds a **strong reference**.  
Strong reference = *"I'm holding this object. Do NOT garbage collect it."*

```
let user = { name: "Alice" };

STACK              HEAP
─────              ──────────────────
user  ──STRONG──▶  { name: "Alice" }
                   kept alive — GC won't touch it
```

A **weak reference** = *"I know where this object is. But if nobody else needs it — go ahead and collect it. Don't keep it alive just for me."*

```
weakRef ····WEAK····▶  { name: "Alice" }
                       NOT kept alive by this reference alone
```

---

### The Problem WeakMap Solves

You want to store data associated with an object (like a cache).  
But you don't want your cache to be the reason the object stays in memory forever.

**With regular Map — LEAKS:**

```js
let cache = new Map();
let user = { name: "Alice" };
cache.set(user, { visits: 5 });

user = null; // you're done with user
```

```
ROOTS               HEAP
─────               ──────────────────────────────
cache ──STRONG──▶   Map {
(global)               key:   { name: "Alice" }  ◀── STRONG ref from Map
                       value: { visits: 5 }
                    }
                    ↑
               Map STILL holds object alive ❌
               Even though user = null
               MEMORY LEAK
```

**With WeakMap — NO LEAK:**

```js
let cache = new WeakMap();
let user = { name: "Alice" };
cache.set(user, { visits: 5 });

user = null; // you're done with user
```

```
ROOTS               HEAP
─────               ──────────────────────────────
cache ····WEAK····▶  WeakMap {
(global)                key:   { name: "Alice" }  ◀·· WEAK ref only
                        value: { visits: 5 }
                     }
                     ↑
                No strong references left
                GC CAN collect the object ✅
                WeakMap entry disappears automatically
```

---

### Map vs WeakMap Side by Side

```
REGULAR MAP                         WEAKMAP
───────────────────────────────     ───────────────────────────────

map.set(obj, "data")                wm.set(obj, "data")

map ──STRONG──▶ obj                 wm ····WEAK····▶ obj

obj = null →                        obj = null →
  map still holds obj ❌              no strong refs left
  LEAK                                GC frees obj ✅
                                      wm entry gone automatically
```

---

### The Problem WeakRef Solves

Sometimes you want to hold a reference to an object — but you don't want to be the reason it stays alive. You just want to use it IF it's still around.

```js
let user = { name: "Alice" };
let weakRef = new WeakRef(user);
```

```
ROOTS               HEAP
─────               ──────────────────
user  ──STRONG──▶   { name: "Alice" }
                          ▲
weakRef ····WEAK··········┘
```

```js
user = null; // strong reference gone
```

```
ROOTS               HEAP
─────               ──────────────────
user = null         { name: "Alice" }
                          ▲
weakRef ····WEAK··········┘
                    ↑
               Only weak ref left
               GC CAN collect this
               Might happen anytime
```

Because the object might be collected at any time, you must **check before using it:**

```js
let deref = weakRef.deref();

if (deref) {
  console.log(deref.name); // ✅ object still alive
} else {
  console.log("Already collected"); // object was GC'd
}
```

```
weakRef.deref():

GC hasn't run yet:              GC already ran:
──────────────────              ────────────────
returns { name: "Alice" }       returns undefined
if (deref) → true ✅            if (deref) → false
use the object                  handle the missing case
```

---

### Map vs WeakMap vs WeakRef — Full Comparison

```
                  REGULAR MAP       WEAKMAP           WEAKREF
                  ───────────       ───────           ───────
What it does      Key-value         Data associated   Reference to
                  storage           with objects      an object

Reference type    STRONG            WEAK              WEAK
                  (prevents GC)     (allows GC)       (allows GC)

Object GC'd       ❌ Never          ✅ When no other  ✅ When no other
when no other     Map holds it      strong refs       strong refs
strong refs?      alive (LEAK)      Entry auto-gone   .deref() → undefined

When to use       General           Cache/metadata    Optional reference —
                  storage           without leaking   use object if alive,
                                                      handle if gone
```

---

### One-Line Interview Answers

**WeakMap:**
> *"A WeakMap holds weak references to its keys. If a key object has no other strong references, it can be garbage collected — the WeakMap entry disappears automatically. Regular Maps prevent this and can cause leaks."*

**WeakRef:**
> *"A WeakRef lets you hold a reference to an object without preventing garbage collection. You use `.deref()` to access it — returns the object if still alive, or `undefined` if already collected."*

## Complete Summary

### Garbage Collection
```
Stack memory   → cleans itself (EC popped = variables gone)
Heap memory    → garbage collector cleans it

Algorithm      → Mark and Sweep
                 Start from roots (global vars, current call stack)
                 Mark everything reachable as alive
                 Sweep (delete) everything not marked

Core concept   → REACHABILITY
                 Reachable from a root = KEEP
                 Not reachable = FREE
```

### The 4 Memory Leak Patterns
```
1. Forgotten setInterval / setTimeout
   → timer keeps callback alive → callback keeps data alive
   → Fix: clearInterval() / clearTimeout() when done

2. Event listener not removed
   → listener is a closure → captures data → data stays alive
   → Fix: removeEventListener() when done

3. Accidental global variable
   → missing let/const/var → goes to window → never cleaned
   → Fix: always use let/const, use "use strict"

4. Detached DOM node
   → removed from page but JS variable still holds reference
   → Fix: set the variable to null after DOM removal
```

### WeakMap / WeakRef
```
Problem they solve:
  You hold a reference to an object
  But you don't want that reference to keep it alive
  = potential memory leak with regular Map/variable

WeakMap  → weak reference to KEYS
           key object GC'd when no other strong refs
           entry disappears automatically
           use case: caching without leaking

WeakRef  → weak reference to ANY object
           object can be GC'd anytime
           use .deref() to check if still alive
           use case: optional reference to large object
```

## Output Questions — Work Through Before Reading Answers

**Q1 — Basic reachability**
```js
let user = { name: "Alice" };
user = null;
// What happens to { name: "Alice" } in memory?
```
<details>
<summary>Answer</summary>

`{ name: "Alice" }` becomes unreachable — no references point to it.  
The garbage collector will free it during the next GC cycle.  
Setting a variable to `null` removes the strong reference, making the object eligible for collection.
</details>

---

**Q2 — Two references**
```js
let a = { x: 1 };
let b = a;
a = null;
// Is { x: 1 } garbage collected?
```
<details>
<summary>Answer</summary>

No. `b` still holds a strong reference to `{ x: 1 }`. The object is still reachable via `b`.  
GC only frees it when ALL references are removed — when `b = null` too.
</details>

---

**Q3 — Stack vs heap cleanup**
```js
function greet() {
  let name = "Alice";
  let user = { age: 25 };
}
greet();
// What happens to name and user after greet() returns?
```
<details>
<summary>Answer</summary>

`name = "Alice"` (primitive on stack) → cleaned up automatically when EC is popped from call stack.  
`user = { age: 25 }` (object on heap) → the reference `user` is gone (EC popped), so `{ age: 25 }` becomes unreachable → GC will free the heap object.
</details>

---

**Q4 — Forgotten timer leak**
```js
function startTimer() {
  let data = { items: new Array(10000) };

  setInterval(function() {
    console.log(data.items.length);
  }, 1000);
}

startTimer();
// Is there a memory leak? Why?
```
<details>
<summary>Answer</summary>

Yes — memory leak.  
The `setInterval` callback holds a reference to `data` via closure.  
As long as the interval runs, `data` is reachable → GC can never free it.  
Fix: save the interval ID and call `clearInterval(id)` when done.
</details>

---

**Q5 — Accidental global**
```js
function foo() {
  bar = "leaked";
}
foo();
// Is there a memory leak? What type?
```
<details>
<summary>Answer</summary>

Yes — accidental global variable leak.  
`bar` has no `let`/`const`/`var` → becomes `window.bar` in browser.  
`window` is always a root → `window.bar` is always reachable → never collected.  
Fix: add `let`/`const`, or use `"use strict"` to catch it as a ReferenceError.
</details>

---

**Q6 — Detached DOM node**
```js
let removed;

function remove() {
  let el = document.getElementById("box");
  document.body.removeChild(el);
  removed = el;
}

remove();
// Is the element freed from memory?
```
<details>
<summary>Answer</summary>

No — detached DOM node leak.  
The element is removed from the visible page, but `removed` (a global variable = root) still holds a reference.  
The element is "detached" — not in DOM, but alive in memory.  
Fix: `removed = null` after removal, or don't store the reference.
</details>

---

**Q7 — WeakMap vs Map**
```js
let map = new Map();
let wm = new WeakMap();
let obj = { name: "Alice" };

map.set(obj, "map data");
wm.set(obj, "wm data");

obj = null;
// What happens in each case?
```
<details>
<summary>Answer</summary>

**Map:** `obj` is gone but `map` still holds a strong reference to the original object. Object cannot be GC'd. Memory leak — the Map entry stays forever.  
**WeakMap:** `obj` is gone. WeakMap only held a weak reference. No strong references remain. Object is eligible for GC. WeakMap entry disappears automatically.
</details>

---

**Q8 — What are roots?**
```js
var globalVar = { x: 1 };

function foo() {
  let localVar = { y: 2 };
  // foo is currently executing
}
```
<details>
<summary>Answer</summary>

`globalVar` → root (global variable, always reachable).  
`localVar` → root WHILE `foo()` is on the call stack.  
After `foo()` returns → `localVar` is no longer a root → `{ y: 2 }` becomes unreachable → GC frees it.
</details>

## What SDE2 Interviewers Expect

**1. Explain GC using reachability**  
*"JS uses reachability to determine what to keep. Starting from roots — global variables and the current call stack — it marks everything reachable as alive. Anything not reachable gets freed. This is the Mark and Sweep algorithm."*

**2. Know what roots are**  
*"Roots are the starting points for reachability — global variables and variables in functions currently on the call stack. Anything reachable from a root is kept alive."*

**3. Know all 4 memory leak patterns with fixes**  
Forgotten timers, event listeners not removed, accidental globals, detached DOM nodes.  
For each: explain why the reference stays alive and how to fix it.

**4. Know the difference between stack and heap cleanup**  
Stack cleans itself when EC is popped. Heap needs GC. Memory leaks only happen on the heap.

**5. Know WeakMap/WeakRef at concept level**  
*"They hold weak references — don't prevent garbage collection. WeakMap for caching data associated with objects without leaking. WeakRef for holding an optional reference to an object that may be collected."*

**6. Connect to real-world symptoms**  
Memory leaks cause: app slowdown over time, browser tab memory growing, eventual crash. Detected via Chrome DevTools Memory/Performance tabs.

## The Mental Models to Carry Into the Interview

**GC = the automatic eraser on a whiteboard.**  
JS writes things on the whiteboard (memory). GC periodically erases anything that's no longer needed. You don't control when it runs.

**Reachability = can you draw a path from a root to the object?**  
If yes → keep. If no → erase. That's the entire GC decision.

**Memory leak = you walked away but left the light on.**  
You're done with the room (object). But you left a reference holding the door open. GC sees the door open and thinks someone's still in there.

**WeakMap/WeakRef = holding an address without owning the house.**  
You know where the object lives. But you're not the owner. If the real owner leaves, the house gets demolished — and that's fine with you.
