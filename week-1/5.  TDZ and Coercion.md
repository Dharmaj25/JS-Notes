## Part 1 тАФ Temporal Dead Zone (TDZ)

### What is TDZ?

`let` and `const` are hoisted тАФ but instead of being initialized to `undefined` like `var`, they are placed in a **locked state** called the Temporal Dead Zone.

TDZ = the period between when a variable is **hoisted** and when its **declaration line is executed**.  
Any access during this period throws a `ReferenceError`.

```
Script starts
     тФВ
     тЦ╝
тФМтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФР
тФВ  CREATION PHASE                 тФВ
тФВ  let b тЖТ hoisted тЖТ enters TDZ  тФВ тЖР b EXISTS but LOCKED тЪая╕П
тФФтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФШ
```

```js
console.log(b); // тЭМ ReferenceError: Cannot access 'b' before initialization
let b = 5;
console.log(b); // тЬЕ 5
```

### Why TDZ Exists

Before `let`/`const`, only `var` existed. `var` gave `undefined` silently тАФ hiding bugs.  
TDZ was introduced to make bugs **loud and immediate**.

```js
console.log(x); // undefined тАФ silent bug, hard to trace (var)
var x = 5;

console.log(y); // ReferenceError тАФ loud, immediate, tells you exactly what's wrong (let)
let y = 5;
```

---

### Proof That let/const ARE Hoisted

People often say *"let/const are not hoisted"*. That's wrong. Here's the proof:

```js
let x = 1;
{
  console.log(x); // тЭМ ReferenceError тАФ NOT 1!
  let x = 2;
}
```

If the inner `let x` was NOT hoisted, `console.log(x)` would look outward and find `x = 1` тЖТ print `1`.  
But the inner `let x` IS hoisted to the top of its block тЖТ enters TDZ тЖТ shadows the outer `x`.  
So `console.log(x)` hits the TDZ variable тЖТ ReferenceError.

**The error message is the proof:**
```
"Cannot access 'x' before initialization"
тЖТ JS knows x EXISTS (it was hoisted), just won't allow access yet

"x is not defined"
тЖТ x doesn't exist anywhere тАФ completely different situation
```

---

### typeof + TDZ тАФ The One Exception

`typeof` is normally 100% safe тАФ never throws, even for undeclared variables:

```js
console.log(typeof undeclaredVar); // "undefined" тАФ no error
```

But in TDZ, `typeof` DOES throw:

```js
console.log(typeof x); // тЭМ ReferenceError: Cannot access 'x' before initialization
let x = 5;
```

> This is the **only exception** to "typeof never throws". TDZ breaks that rule.

---

### Function Default Parameters + TDZ

```js
function foo(a = b, b = 2) {
  return a + b;
}
foo(); // тЭМ ReferenceError
```

Parameters are evaluated **left to right**. When `a = b` evaluates, `b` is still in TDZ.

```js
// Fix тАФ swap the order
function foo(a = 2, b = a) {
  return a + b;
}
foo(); // тЬЕ 4
```

## Part 2 тАФ Type Coercion

### What is Type Coercion?

JavaScript is **loosely typed** тАФ you never declare what type a variable is.  
When you do an operation between two different types, JS **automatically converts** one type to another.  
This automatic conversion is called **type coercion**.

```js
"5" + 1   // "51"  тЖР number converted to string
"5" - 1   // 4     тЖР string converted to number
true + 1  // 2     тЖР boolean converted to number
```

---

### Implicit vs Explicit Coercion

**Implicit** тАФ JS does it automatically:
```js
"5" * 2    // 10   тЖР JS converted "5" to number
if (0) {}  // 0 coerced to false
```

**Explicit** тАФ you do it yourself:
```js
Number("5")   // 5
String(42)    // "42"
Boolean(0)    // false
```

> Interviews mostly test **implicit coercion** тАФ that's where the tricky output questions live.

---

### Converting to Number тАФ The Rules

```js
Number("42")       // 42      тЖР valid number string
Number("3.14")     // 3.14
Number("")         // 0       тЖР empty string тЖТ 0
Number("  ")       // 0       тЖР whitespace only тЖТ 0
Number("abc")      // NaN     тЖР invalid
Number("12abc")    // NaN     тЖР can't fully parse
Number(true)       // 1
Number(false)      // 0
Number(null)       // 0       тЖР null тЖТ 0
Number(undefined)  // NaN     тЖР undefined тЖТ NaN
Number([])         // 0       тЖР empty array тЖТ 0
Number([3])        // 3       тЖР single item array тЖТ that number
Number([1,2])      // NaN     тЖР multiple items тЖТ NaN
Number({})         // NaN     тЖР object тЖТ NaN
```

> **Must memorize:** `null тЖТ 0`, `undefined тЖТ NaN`, `"" тЖТ 0`, `true тЖТ 1`, `false тЖТ 0`

---

### Converting to String тАФ The Rules

```js
String(42)         // "42"
String(3.14)       // "3.14"
String(true)       // "true"
String(false)      // "false"
String(null)       // "null"
String(undefined)  // "undefined"
String([1,2,3])    // "1,2,3"
String({})         // "[object Object]"
```

---

### Converting to Boolean тАФ Truthy and Falsy

Every value in JS is either **truthy** or **falsy** when converted to boolean.

#### The 8 Falsy Values тАФ Memorize All of These

```js
Boolean(false)     // false
Boolean(0)         // false
Boolean(-0)        // false
Boolean(0n)        // false  тЖР BigInt zero
Boolean("")        // false  тЖР empty string only
Boolean(null)      // false
Boolean(undefined) // false
Boolean(NaN)       // false
```

> **Memory trick:** `false, 0, -0, 0n, "", null, undefined, NaN` тАФ that's the complete list.  
> **Everything else is truthy.**

#### The Truthy Traps тАФ These Catch Everyone

```js
Boolean("0")       // true  тЖР string "0" is NOT falsy (not empty string)
Boolean("false")   // true  тЖР non-empty string, always truthy
Boolean(" ")       // true  тЖР space is NOT empty string
Boolean([])        // true  тЖР empty array is TRUTHY тЖР very common trap
Boolean({})        // true  тЖР empty object is TRUTHY тЖР very common trap
Boolean(-1)        // true  тЖР any non-zero number is truthy
Boolean(Infinity)  // true
Boolean([0])       // true  тЖР array with falsy value inside is still truthy
Boolean(function(){}) // true
```

> **The two biggest traps:** `[]` and `{}` are both **truthy** even though they're "empty".

## Part 3 тАФ == vs ===

### === (Strict Equality) тАФ No Coercion, Ever

Checks **value AND type**. If types differ тЖТ immediately `false`. No conversion happens.

```js
5 === 5           // true  тЖР same type, same value
5 === "5"         // false тЖР different types тЖТ false immediately
null === null     // true
null === undefined // false тЖР different types
false === 0       // false тЖР different types
```

---

### == (Loose Equality) тАФ Coerces Types First

Checks value but **converts types** if they differ. Follows a specific set of rules.

---

### The == Rules тАФ In Order

#### Rule 1 тАФ Same type? Compare directly (same as ===)

```js
5 == 5       // true
"hi" == "hi" // true
```

---

#### Rule 2 тАФ null and undefined: only equal to each other

```js
null == undefined  // true  тЖР special pair, hardcoded rule
null == 0          // false тЖР null doesn't coerce to 0 in ==
null == ""         // false
null == false      // false
undefined == 0     // false
undefined == false // false
```

> `null` and `undefined` only `==` each other. Nothing else. No exceptions.

---

#### Rule 3 тАФ NaN: never equal to anything, including itself

```js
NaN == NaN   // false тЖР always false, no exceptions
NaN === NaN  // false
```

> To check for NaN: use `Number.isNaN(x)` or `x !== x` (only NaN fails self-equality check).

---

#### Rule 4 тАФ Number vs String тЖТ convert string to number, then compare

```js
5 == "5"    // true  тЖР "5" тЖТ 5 тЖТ 5 == 5
0 == ""     // true  тЖР "" тЖТ 0 тЖТ 0 == 0
0 == "0"    // true  тЖР "0" тЖТ 0 тЖТ 0 == 0
1 == "1"    // true
2 == "2"    // true
```

---

#### Rule 5 тАФ Boolean vs anything тЖТ convert boolean to number FIRST, then reapply rules

```js
true == 1      // true  тЖР true тЖТ 1 тЖТ 1 == 1
false == 0     // true  тЖР false тЖТ 0 тЖТ 0 == 0
true == "1"    // true  тЖР true тЖТ 1, "1" тЖТ 1 тЖТ 1 == 1
true == "2"    // false тЖР true тЖТ 1, "2" тЖТ 2 тЖТ 1 == 2
false == ""    // true  тЖР false тЖТ 0, "" тЖТ 0 тЖТ 0 == 0
false == "0"   // true  тЖР false тЖТ 0, "0" тЖТ 0 тЖТ 0 == 0
false == null  // false тЖР false тЖТ 0, null stays (Rule 2 takes over)
false == undefined // false
```

> **Key:** boolean always converts to number first. THEN the comparison restarts from the top.

---

#### Rule 6 тАФ Object vs primitive тЖТ convert object to primitive, then reapply rules

JS calls `.valueOf()` then `.toString()` on the object.

```js
[] == 0        // true  тЖР [] тЖТ "" тЖТ 0 тЖТ 0 == 0
[] == ""       // true  тЖР [] тЖТ "" тЖТ "" == ""
[] == false    // true  тЖР false тЖТ 0, [] тЖТ "" тЖТ 0 тЖТ 0 == 0
[1] == 1       // true  тЖР [1] тЖТ "1" тЖТ 1 тЖТ 1 == 1
[1] == "1"     // true  тЖР [1] тЖТ "1" тЖТ "1" == "1"
[[]] == 0      // true  тЖР [[]] тЖТ "" тЖТ 0 тЖТ 0 == 0
```

---

### The Most Famous == Trap тАФ `[] == ![]`

```js
[] == ![]  // true тАФ the most surprising result in JS
```

Step by step:
```
Step 1: Evaluate ![] first
        [] is truthy тЖТ ![] = false

Step 2: [] == false
        false is boolean тЖТ convert to number тЖТ 0
        Now: [] == 0

Step 3: [] is object, 0 is number
        Convert [] to primitive тЖТ [] тЖТ "" (empty string)
        Now: "" == 0

Step 4: "" is string, 0 is number
        Convert "" to number тЖТ 0
        Now: 0 == 0 тЖТ true тЬЕ
```

> If you can explain this step by step in an interview, you will genuinely impress the interviewer.

---

### == Decision Flowchart

```
a == b  (types are different)
тФВ
тФЬтФА Is one null and the other undefined?
тФВ   тФФтФА YES тЖТ true (special pair)
тФВ
тФЬтФА Is either value NaN?
тФВ   тФФтФА YES тЖТ false (always)
тФВ
тФЬтФА Is one a boolean?
тФВ   тФФтФА YES тЖТ convert boolean to number тЖТ restart comparison
тФВ
тФЬтФА Is one a string and other a number?
тФВ   тФФтФА YES тЖТ convert string to number тЖТ compare
тФВ
тФЬтФА Is one an object and other a primitive?
тФВ   тФФтФА YES тЖТ convert object to primitive тЖТ restart comparison
тФВ
тФФтФА Otherwise тЖТ false
```

---

### == vs === Summary

```
===  тЖТ  No coercion. Types differ = false. Always predictable.
==   тЖТ  Coerces types. Follows 6 rules. Often surprising.

Use ===  тЖТ  always, by default
Use ==   тЖТ  only for null/undefined check:

if (value == null) {
  // catches BOTH null AND undefined
  // cleaner than: value === null || value === undefined
}
```

## Part 4 тАФ Coercion in Operators

### + operator тАФ prefers string (if any operand is a string)

```js
1 + "2"         // "12"  тЖР number тЖТ string тЖТ concatenate
"3" + 4 + 5     // "345" тЖР left to right: "3"+4="34", "34"+5="345"
3 + 4 + "5"     // "75"  тЖР left to right: 3+4=7, 7+"5"="75"
true + true     // 2     тЖР 1 + 1
true + true + "1" // "21" тЖР 1+1=2, then 2+"1"="21" (STRING, not number 21)
1 + null        // 1     тЖР null тЖТ 0
1 + undefined   // NaN   тЖР undefined тЖТ NaN
```

> **The `+` trap:** `true + true + "1"` = `"21"` (string) not `21` (number).  
> Arithmetic happens first (2), then string concatenation kicks in ("21").

---

### -, *, / operators тАФ always convert to number

```js
"5" - 2       // 3    тЖР "5" тЖТ 5
"5" * "2"     // 10   тЖР both тЖТ number
null - 1      // -1   тЖР null тЖТ 0
undefined - 1 // NaN  тЖР undefined тЖТ NaN
true - false  // 1    тЖР 1 - 0
"" - 1        // -1   тЖР "" тЖТ 0
```

---

### Unary + тАФ quick way to convert to number

```js
+"42"       // 42
+""         // 0
+true       // 1
+false      // 0
+null       // 0
+undefined  // NaN
+[]         // 0
+{}         // NaN
+"abc"      // NaN
```

## Part 5 тАФ Logical Operators + Coercion

`&&` and `||` don't return `true`/`false` тАФ they return one of the **actual operand values**.

### || (OR) тАФ returns first TRUTHY value, or last value if all falsy

```js
"hello" || "world"    // "hello" тЖР first truthy, stops here
0 || "world"          // "world" тЖР 0 is falsy, moves on
null || undefined     // undefined тЖР both falsy, returns last
false || 0 || "yes"   // "yes"   тЖР first truthy found
0 || ""               // ""      тЖР both falsy, returns last
```

### && (AND) тАФ returns first FALSY value, or last value if all truthy

```js
"hello" && "world"    // "world" тЖР both truthy, returns last
0 && "world"          // 0       тЖР first falsy, stops here
"yes" && 42           // 42      тЖР both truthy, returns last
null && "anything"    // null    тЖР first falsy, stops here
1 && 2 && 3           // 3       тЖР all truthy, returns last
```

---

### ?? (Nullish Coalescing) vs || (OR) тАФ Critical Difference

```js
// || falls back for ALL falsy values (0, "", false, null, undefined, NaN)
const a = 0 || "default"    // "default" тЖР 0 is falsy
const b = "" || "default"   // "default" тЖР "" is falsy

// ?? only falls back for null and undefined тАФ nothing else
const c = 0 ?? "default"    // 0   тЖР 0 is not null/undefined, keeps it
const d = "" ?? "default"   // ""  тЖР kept
const e = null ?? "default" // "default" тЖР null triggers fallback
const f = undefined ?? "default" // "default" тЖР undefined triggers fallback
```

> **Use `??` when `0` or `""` are valid values you want to keep.**  
> **Use `||` when you want to fall back on any falsy value.**

**Q1 тАФ == with boolean and number**

<details>
<summary>Answer</summary>

</details>

<summary>Answer</summary>

</details>

**Q12 тАФ || and && return values**

`"c"` тАФ all truthy, `&&` returns last value.
</details>

## What SDE2 Interviewers Expect

**1. Know the 8 falsy values without hesitation**  
`false, 0, -0, 0n, "", null, undefined, NaN`. Everything else is truthy тАФ especially `[]`, `{}`, `"0"`, `"false"`.

**2. Know the == rules in order**  
null/undefined special pair тЖТ NaN always false тЖТ boolean converts to number тЖТ string vs number converts string тЖТ object vs primitive converts object. Explain step by step.

**3. Always use === unless you have a specific reason**  
The one valid use of `==`: `value == null` catches both `null` and `undefined` cleanly.

**4. Explain [] == ![] step by step**  
This is the ultimate coercion question. Four steps. If you can walk through it, you know coercion cold.

**5. Know TDZ breaks typeof**  
typeof normally never throws. TDZ is the one exception. Very common follow-up.

**6. Know ?? vs || difference**  
`||` falls back on any falsy. `??` only falls back on `null`/`undefined`. Real-world usage: prefer `??` for default values when `0` or `""` are valid.

**7. Know + operator left-to-right evaluation**  
`1 + 2 + "3"` = `"33"`. `"1" + 2 + 3` = `"123"`. The position of the string determines when concatenation kicks in.

**8. Know && and || return actual values, not booleans**  
`||` returns first truthy or last value. `&&` returns first falsy or last value. This is how default value patterns work.

## The Mental Models to Carry Into the Interview

**Coercion = JS trying to make the operation work.**  
It has a rulebook. Learn the rulebook and coercion becomes completely predictable.

**=== = strict bouncer. Checks ID (type) before anything.**  
Wrong type? You're not getting in. No exceptions.

**== = lenient bouncer. Will convert your ID first if needed.**  
Follows a specific set of rules to convert. Know the rules, know the result.

**Falsy = the 8 outsiders. Everything else is truthy.**  
The two biggest surprises: `[]` and `{}` are truthy. `"0"` is truthy. Memorize the 8.

**TDZ = a locked door with a sign.**  
JS put the variable behind the door during creation phase. The sign says "not ready yet". The door unlocks when the declaration line runs.
