# Execution Context + Call Stack + Lexical Environment — SDE2 Interview Reference.
---

## The Big Picture First

When you write JS and run it — what actually happens?

Most people know bits. Hoisting happens. Scope exists. Variables get values. But how does it all fit together?

The answer is: **Execution Context**.

Every variable, every function call, every scope — all of it is managed through Execution Contexts. This is the engine that runs your JS code. Once you understand this, everything else — hoisting, scope chain, TDZ, closures — stops feeling like magic and becomes a machine you fully understand.

---

## What is an Execution Context?

An Execution Context is a **box** JS creates every time it runs some code.

Inside that box:
- All the variables for that piece of code
- The value of `this`
- A reference to the outer scope (for scope chain)

```
┌─────────────────────────────────────────┐
│           EXECUTION CONTEXT             │
│                                         │
│  Variable Environment                   │
│  (var declarations live here)           │
│                                         │
│  Lexical Environment                    │
│  (let/const live here + outer ref)      │
│                                         │
│  this → (some value)                    │
└─────────────────────────────────────────┘
```

---

## Two Types of Execution Context

### 1. Global Execution Context (GEC)

- Created **once** when your script first loads
- There is only ever **ONE** GEC
- Represents everything outside all functions
- `this` = `window` (browser) / `global` (Node.js)

```js
var name = "Alice";    // lives in GEC
let age = 25;          // lives in GEC
function greet() {}    // greet reference lives in GEC
                       // calling greet() creates a NEW context
```

### 2. Function Execution Context (FEC)

- Created **every time a function is called** (not when defined — when called)
- Each call = fresh box, fresh variables
- `this` depends on how the function is called (Week 2 topic)

```js
function greet() {
  var message = "Hello";
}

greet(); // new FEC created
greet(); // another new FEC created — completely fresh
greet(); // and another
```

---

## The Two Phases — Connected to Hoisting

Every execution context (GEC or FEC) runs in two phases.  
This is exactly what you already know as **hoisting**.

### Phase 1 — Creation Phase

Before a single line of code runs, JS sets up the context:

```
var declarations     → stored as undefined
let/const            → stored but locked in TDZ ⚠️
function declarations → entire body stored in memory ✅
parameters           → set to passed values
this                 → assigned
outer reference      → set based on where code was written (lexical scope)
```

### Phase 2 — Execution Phase

Code runs line by line:
```
Assignments happen (var gets its real value)
Functions get called → new FEC created and pushed to call stack
let/const exit TDZ when their declaration line is reached
Function declaration lines are skipped (already done in Phase 1)
```

> **Hoisting IS the creation phase.** Same concept, two names for the same thing.

## The Call Stack — Managing Multiple Contexts

Every function call creates a new execution context.  
JS needs to track all of them. It uses the **Call Stack**.

### What is the Call Stack?

A **stack data structure** — last in, first out (like a stack of plates).  
JS uses it to track which execution context is currently running.

> **Rule: Whatever is on TOP of the stack is currently executing.**

---

### Call Stack — Step by Step Walkthrough

```js
function multiply(a, b) {
  return a * b;
}

function square(n) {
  return multiply(n, n);
}

function printSquare(n) {
  var result = square(n);
  console.log(result);
}

printSquare(4);
```

**Step 1 — Script loads. GEC created and pushed.**
```
CALL STACK
┌──────────────────────┐
│  Global EC           │  ← currently executing
└──────────────────────┘

Creation phase: multiply, square, printSquare all stored as functions
```

**Step 2 — `printSquare(4)` called. New FEC pushed on top.**
```
CALL STACK
┌──────────────────────┐
│  printSquare(4) EC   │  ← currently executing
├──────────────────────┤
│  Global EC           │  ← waiting
└──────────────────────┘

Creation phase: n = 4, result = undefined
```

**Step 3 — Inside `printSquare`, `square(4)` called. New FEC pushed.**
```
CALL STACK
┌──────────────────────┐
│  square(4) EC        │  ← currently executing
├──────────────────────┤
│  printSquare(4) EC   │  ← waiting
├──────────────────────┤
│  Global EC           │  ← waiting
└──────────────────────┘
```

**Step 4 — Inside `square`, `multiply(4, 4)` called. New FEC pushed.**
```
CALL STACK
┌──────────────────────┐
│  multiply(4, 4) EC   │  ← currently executing
├──────────────────────┤
│  square(4) EC        │  ← waiting
├──────────────────────┤
│  printSquare(4) EC   │  ← waiting
├──────────────────────┤
│  Global EC           │  ← waiting
└──────────────────────┘
```

**Step 5 — `multiply` returns 16. EC popped.**
```
CALL STACK
┌──────────────────────┐
│  square(4) EC        │  ← resumes, gets 16 back
├──────────────────────┤
│  printSquare(4) EC   │  ← waiting
├──────────────────────┤
│  Global EC           │  ← waiting
└──────────────────────┘
```

**Step 6 — `square` returns 16. EC popped.**
```
CALL STACK
┌──────────────────────┐
│  printSquare(4) EC   │  ← resumes, result = 16, console.log runs
├──────────────────────┤
│  Global EC           │  ← waiting
└──────────────────────┘
```

**Step 7 — `printSquare` finishes. EC popped.**
```
CALL STACK
┌──────────────────────┐
│  Global EC           │  ← back to global
└──────────────────────┘
```

**Step 8 — Script done. GEC popped. Stack empty.**
```
CALL STACK
└──────────────────────┘  (empty)
```

---

### The Call Stack Rules

```
1. GEC always at the bottom — created first, removed last
2. Every function CALL  → pushes a new EC on top
3. Every function RETURN → pops its EC off
4. Whatever is on TOP is what's currently running
5. JS is single-threaded — only ONE thing runs at a time
```

---

## Stack Overflow

```js
function infinite() {
  infinite(); // calls itself forever, no base case
}
infinite(); // ❌ RangeError: Maximum call stack size exceeded
```

Every call pushes a new EC. No return = no pop. Stack fills to its limit and crashes.

```
CALL STACK (fills to limit)
┌──────────────────────┐
│  infinite() EC       │
├──────────────────────┤
│  infinite() EC       │
├──────────────────────┤
│  infinite() EC       │
├──────────────────────┤
│  ... thousands more  │
├──────────────────────┤
│  Global EC           │
└──────────────────────┘
         💥 RangeError: Maximum call stack size exceeded
```

> Always ensure recursive functions have a base case that stops the recursion.

## Lexical Environment — The Deep Dive

Every execution context has a **Lexical Environment** — the actual internal data structure JS uses to manage scope.

### A Lexical Environment has two parts:

```
┌──────────────────────────────────────────────────┐
│              LEXICAL ENVIRONMENT                 │
│                                                  │
│  1. Environment Record                           │
│     → stores all variables and functions         │
│       declared in this scope                     │
│                                                  │
│  2. Outer Reference                              │
│     → points to the PARENT lexical environment  │
│     → this is what makes the scope chain work   │
│                                                  │
└──────────────────────────────────────────────────┘
```

---

### How Lexical Environments Enable the Scope Chain

```js
var a = 1;

function outer() {
  var b = 2;

  function inner() {
    var c = 3;
    console.log(a + b + c); // how does inner find a and b?
  }

  inner();
}

outer();
```

Here is what the Lexical Environments look like in memory:

```
┌──────────────────────────────────────────────────────┐
│  GLOBAL LEXICAL ENVIRONMENT                          │
│                                                      │
│  Environment Record:                                 │
│    a = 1                                             │
│    outer = function                                  │
│                                                      │
│  Outer Reference: null  (nothing above global)       │
└──────────────────────────────────────────────────────┘
              ▲
              │ outer reference points here
              │
┌──────────────────────────────────────────────────────┐
│  outer() LEXICAL ENVIRONMENT                         │
│                                                      │
│  Environment Record:                                 │
│    b = 2                                             │
│    inner = function                                  │
│                                                      │
│  Outer Reference: → Global LE ↑                      │
└──────────────────────────────────────────────────────┘
              ▲
              │ outer reference points here
              │
┌──────────────────────────────────────────────────────┐
│  inner() LEXICAL ENVIRONMENT                         │
│                                                      │
│  Environment Record:                                 │
│    c = 3                                             │
│                                                      │
│  Outer Reference: → outer() LE ↑                    │
└──────────────────────────────────────────────────────┘
```

When `inner` looks up `a + b + c`:
- `c` → found in own Environment Record ✅ stops here
- `b` → not here → follow outer ref → found in `outer()` LE ✅ stops here
- `a` → not here → follow outer ref → not in `outer()` → follow its outer ref → found in Global LE ✅

**The chain of outer references = the scope chain.**  
**This is literally what the scope chain is — a linked list of Lexical Environments.**

## Variable Environment vs Lexical Environment — The Important Distinction

This is SDE2-level detail. Most candidates don't know this.

### At the Global Level — They Are the SAME Object

```
GLOBAL EXECUTION CONTEXT

  Variable Environment  ──┐
                           ├──▶  { x: "global", foo: fn, bar: fn }
  Lexical Environment   ──┘

  Both point to the SAME environment record.
```

This means `var` declarations at global scope ARE accessible through the Lexical Environment chain — because there's no separation at global level.

```js
var x = "global";   // → GEC's Variable Environment
                    // = GEC's Lexical Environment (same object)

function foo() {
  console.log(x);   // foo's outer ref → GEC's Lexical Env
                    // x found there ✅
                    // because at global scope, VE = LE
}
```

### Inside Functions and Blocks — They Separate

```
FUNCTION EXECUTION CONTEXT

  Variable Environment  ──▶  { var declarations — function scoped }

  Lexical Environment   ──▶  { let/const declarations — block scoped }
                              + outer reference → parent LE
```

```js
function outer() {
  var a = 1;    // → Variable Environment of outer's EC
  let b = 2;    // → Lexical Environment of outer's EC

  {
    let c = 3;  // → NEW block-level Lexical Environment
    var d = 4;  // → still outer's Variable Environment (var ignores block)
  }

  console.log(a); // ✅ 1
  console.log(b); // ✅ 2
  console.log(c); // ❌ ReferenceError — block LE is gone
  console.log(d); // ✅ 4 — in outer's Variable Env, still alive
}
```

### The Full Table

```
SCOPE LEVEL       Variable Environment      Lexical Environment
──────────────────────────────────────────────────────────────────
Global scope      Same object  ◀──────────▶ Same object
Function scope    var lives here             let/const live here
Block scope       does not exist             let/const live here
```

> **This is why `var` leaks out of blocks but `let`/`const` don't.**  
> `var` lives in the Variable Environment which is function-scoped.  
> `let`/`const` live in Lexical Environments which are block-scoped.

## Lexical Scope — Outer Reference is Set at WRITE Time

The outer reference is set based on **where the function was written** — not where it is called from.

```js
var x = "global";

function foo() {
  console.log(x); // foo's outer ref → GEC (where foo was WRITTEN)
}

function bar() {
  var x = "bar";
  foo();           // foo CALLED from here — doesn't matter
}

bar(); // prints "global" — not "bar"
```

`foo` was written in global scope → its outer reference points to GEC → it finds `x = "global"`.  
`bar`'s `x` is in `bar`'s own LE → `foo` never looks there → invisible to `foo`.

> **Outer reference = set at write time. Never changes. Ever.**  
> This is lexical scope. This is also the foundation of closures (Week 2).

## The Complete Machine — Everything Together

```
┌──────────────────────────────────────────────────────────────────────┐
│                      THE COMPLETE JS ENGINE                          │
│                                                                      │
│  Script loads                                                        │
│       │                                                              │
│       ▼                                                              │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │  GLOBAL EXECUTION CONTEXT created, pushed to call stack     │    │
│  │                                                             │    │
│  │  CREATION PHASE (hoisting):                                 │    │
│  │    var       → undefined                                    │    │
│  │    let/const → TDZ                                          │    │
│  │    fn decl   → fully stored                                 │    │
│  │    this      → window / global                              │    │
│  │    outer ref → null                                         │    │
│  │                                                             │    │
│  │  EXECUTION PHASE (code runs line by line):                  │    │
│  │    assignments happen                                       │    │
│  │    function called?                                         │    │
│  │         │                                                   │    │
│  │         ▼                                                   │    │
│  │    ┌─────────────────────────────────────────────────┐     │    │
│  │    │  NEW FUNCTION EC created, pushed to call stack  │     │    │
│  │    │                                                 │     │    │
│  │    │  CREATION PHASE:                                │     │    │
│  │    │    parameters set                               │     │    │
│  │    │    var → undefined                              │     │    │
│  │    │    let/const → TDZ                              │     │    │
│  │    │    outer ref → set (where fn was WRITTEN)       │     │    │
│  │    │                                                 │     │    │
│  │    │  EXECUTION PHASE:                               │     │    │
│  │    │    code runs                                    │     │    │
│  │    │    nested calls → more ECs pushed               │     │    │
│  │    │    return → EC POPPED from call stack           │     │    │
│  │    │    control returns to caller                    │     │    │
│  │    └─────────────────────────────────────────────────┘     │    │
│  └─────────────────────────────────────────────────────────────┘    │
│                                                                      │
│  SCOPE CHAIN  = chain of outer references in Lexical Environments   │
│  HOISTING     = what happens during creation phase                  │
│  TDZ          = let/const state during creation phase               │
│  CLOSURES     = fn keeping its Lexical Environment alive (Week 2)   │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

## Output Questions — Work Through Before Reading Answers

**Q1 — Creation phase order**
```js
console.log(a);
console.log(foo);
var a = 1;
function foo() { return 2; }
```
<details>
<summary>Answer</summary>

`undefined` — `var a` hoisted as undefined in creation phase.  
`ƒ foo() { return 2; }` — function declaration fully stored in creation phase.  
Function declarations win over var in hoisting priority.
</details>

---

**Q2 — Call stack order**
```js
function first() {
  console.log("first");
  second();
  console.log("first again");
}
function second() {
  console.log("second");
}
first();
```
<details>
<summary>Answer</summary>

`"first"`, `"second"`, `"first again"`.  
`first` EC pushed → logs "first" → `second` EC pushed → logs "second" → `second` EC popped → back in `first` → logs "first again" → `first` EC popped.
</details>

---

**Q3 — Stack overflow**
```js
function recurse() {
  recurse();
}
recurse();
```
<details>
<summary>Answer</summary>

`RangeError: Maximum call stack size exceeded`.  
Infinite recursion — each call pushes a new EC with no return to pop it. Stack fills up and crashes.
</details>

---

**Q4 — Lexical scope vs call site**
```js
var x = "global";
function foo() {
  console.log(x);
}
function bar() {
  var x = "bar";
  foo();
}
bar();
```
<details>
<summary>Answer</summary>

`"global"`.  
`foo`'s outer reference points to GEC — where `foo` was **written**. Calling `foo` from inside `bar` doesn't change its outer reference. `bar`'s `x` is invisible to `foo`. This is lexical scope.
</details>

---

**Q5 — Each function call gets its own EC**
```js
function counter() {
  var count = 0;
  count++;
  console.log(count);
}
counter();
counter();
counter();
```
<details>
<summary>Answer</summary>

`1`, `1`, `1`.  
Each call to `counter` creates a fresh EC. `count` starts at `0` every time. Variables don't persist between calls unless closures are used (Week 2).
</details>

---

**Q6 — Scope chain lookup through lexical environments**
```js
var a = 1;
function outer() {
  var b = 2;
  function inner() {
    var c = 3;
    console.log(a + b + c);
  }
  inner();
}
outer();
```
<details>
<summary>Answer</summary>

`6`.  
`c` found in `inner`'s own LE. `b` found in `outer`'s LE via outer reference. `a` found in GEC's LE via outer reference chain. Scope chain in action.
</details>

---

**Q7 — var vs let in block — Variable Env vs Lexical Env**
```js
function test() {
  if (true) {
    var a = 1;
    let b = 2;
  }
  console.log(a);
  console.log(b);
}
test();
```
<details>
<summary>Answer</summary>

`1` — `var a` lives in `test`'s Variable Environment (function scoped), accessible anywhere in the function.  
`ReferenceError: b is not defined` — `let b` lives in the block's Lexical Environment, which no longer exists after the block ends.
</details>

---

**Q8 — Hoisting inside a function (not just global)**
```js
function test() {
  console.log(x);
  console.log(y);
  var x = 10;
  let y = 20;
}
test();
```
<details>
<summary>Answer</summary>

`undefined` — `var x` hoisted to top of `test`'s EC creation phase, initialized to `undefined`.  
`ReferenceError` — `let y` hoisted but in TDZ.  
Hoisting happens in EVERY execution context creation phase — not just global.
</details>

---

**Q9 — Function EC is fresh every call**
```js
function add(a, b) {
  var result = a + b;
  return result;
}
console.log(add(2, 3));
console.log(add(10, 20));
```
<details>
<summary>Answer</summary>

`5`, then `30`.  
Each call creates a completely fresh EC with its own `a`, `b`, and `result`. No state shared between calls.
</details>

---

**Q10 — The complete picture question**
```js
var x = 1;

function foo() {
  var x = 2;
  function bar() {
    console.log(x);
  }
  bar();
}

foo();
console.log(x);
```
<details>
<summary>Answer</summary>

`2`, then `1`.  
`bar` was written inside `foo` → its outer reference points to `foo`'s LE → finds `x = 2` there.  
`bar` never reaches GEC for `x` because it found it in `foo`'s LE first.  
Global `x = 1` is untouched.  
Three ECs created: GEC, `foo` EC, `bar` EC. Each with their own environment records.
</details>

## What SDE2 Interviewers Expect

**1. Explain the two types of execution context**  
GEC (once, at startup) and FEC (every function call). Each has creation + execution phase. GEC lives until script ends. FEC lives until function returns.

**2. Explain the call stack clearly**  
*"A LIFO stack that tracks execution contexts. GEC is always at the bottom. Function calls push new ECs on top. Returns pop them off. JS is single-threaded — only the top EC runs at any time."*

**3. Connect hoisting to creation phase**  
*"Hoisting is just the creation phase of an execution context — where JS allocates memory for declarations before running code."*

**4. Explain lexical environment and scope chain**  
*"Each execution context has a Lexical Environment with an Environment Record (stores variables) and an Outer Reference (points to parent LE). The scope chain is this chain of outer references."*

**5. Know that outer reference is set at write time**  
*"The outer reference is set based on where the function is written — not where it's called. This is lexical scope. `foo` defined in global always has GEC as its outer, no matter where it's called from."*

**6. Know the Variable Env vs Lexical Env distinction**  
*"At global scope, they're the same object. Inside functions/blocks they differ — `var` goes into Variable Environment (function-scoped), `let`/`const` go into Lexical Environment (block-scoped). This is why `var` leaks out of blocks."*

**7. Connect everything to closures (preview)**  
*"When a function returns, its EC is popped. But if an inner function was returned, it keeps a reference to its outer Lexical Environment. That environment stays alive in memory. That's a closure."*  
This is the natural lead-in to Week 2.

## The Mental Models to Carry Into the Interview

**Execution Context = a box JS creates for every piece of code.**  
GEC for the script. A fresh FEC for every function call. Each box has its own variables, `this`, and a pointer to its parent.

**Call Stack = the to-do pile.**  
JS can only do one thing at a time. The call stack tracks what's running (top) and what's waiting. Push to run, pop when done.

**Lexical Environment = the box's address book.**  
Environment Record = what's stored here. Outer Reference = where to look if it's not here. Follow the chain outward until found or ReferenceError.

**Outer reference = set the moment the function is written. Never changes.**  
Called from somewhere else? Doesn't matter. The outer reference is frozen at definition time. This is why lexical scope works the way it does.

**Variable Env vs Lexical Env = two filing systems.**  
At global level, same filing cabinet. Inside functions/blocks, `var` goes in one drawer (function-scoped), `let`/`const` go in another (block-scoped). That's why they behave differently.
