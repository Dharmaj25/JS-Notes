## What is Scope?

Scope answers one simple question:

> **"Where in your code can this variable be accessed?"**

That's it. Scope = the area of your code where a variable **exists** and can be **used**.

Think of it like rooms in a house:
- A variable is an object kept in a room
- You can always use it **inside** that room
- Whether you can use it **outside** depends on which room it's in

---

## The 3 Types of Scope

### 1. Global Scope

Any variable declared **outside** all functions and blocks.  
Accessible **everywhere** in your code.

```js
var name = "Alice";
let age = 25;
const city = "Jaipur";

function greet() {
  console.log(name);  // ✅ accessible inside function
  console.log(age);   // ✅ accessible
  console.log(city);  // ✅ accessible
}

greet();
console.log(name);    // ✅ accessible outside too
```

> Think of global scope as the **living room** — everyone in the house can access it.

---

### 2. Function Scope

Variables declared **inside a function** only exist inside that function.  
They are created when the function runs and destroyed when it finishes.

```js
function greet() {
  var message = "Hello!";
  console.log(message);  // ✅ works
}

greet();
console.log(message);    // ❌ ReferenceError: message is not defined
```

> Think of function scope as a **bedroom** — only accessible from inside that room.

---

### 3. Block Scope

Variables declared with `let` or `const` inside any `{}` only exist inside that block.

```js
{
  let x = 10;
  const y = 20;
  console.log(x);  // ✅
}

console.log(x);    // ❌ ReferenceError
console.log(y);    // ❌ ReferenceError
```

`if`, `for`, `while`, `switch` — all create blocks:

```js
if (true) {
  let secret = "hidden";
  var leaked = "I escaped!";
}

console.log(secret);  // ❌ ReferenceError — let is block scoped
console.log(leaked);  // ✅ "I escaped!" — var is NOT block scoped
```

> **Critical:** `var` does NOT respect block scope. Only `let` and `const` do.  
> `var` only respects **function** scope.

## Nested Scope — Scopes Inside Scopes

Functions and blocks can live inside other functions and blocks.  
This creates **nested scopes** — layers inside layers.

```js
var global = "global";

function outer() {
  var outerVar = "outer";

  function inner() {
    var innerVar = "inner";

    console.log(innerVar);  // ✅ own scope
    console.log(outerVar);  // ✅ parent scope
    console.log(global);    // ✅ grandparent scope
  }

  inner();
  console.log(innerVar);    // ❌ cannot see into child
}

outer();
console.log(outerVar);      // ❌ cannot see into function
```

**The golden rule:**

> Inner scopes can look **outward**. Outer scopes can NEVER look **inward**.

```
┌────────────────────────────────────────────┐
│  GLOBAL SCOPE                              │
│  global = "global"                         │
│                                            │
│  ┌──────────────────────────────────────┐  │
│  │  OUTER FUNCTION SCOPE                │  │
│  │  outerVar = "outer"                  │  │
│  │                                      │  │
│  │  ┌────────────────────────────────┐  │  │
│  │  │  INNER FUNCTION SCOPE          │  │  │
│  │  │  innerVar = "inner"            │  │  │
│  │  │                                │  │  │
│  │  │  sees: innerVar  ✅            │  │  │
│  │  │  sees: outerVar  ✅  (outward) │  │  │
│  │  │  sees: global    ✅  (outward) │  │  │
│  │  └────────────────────────────────┘  │  │
│  │                                      │  │
│  │  sees: outerVar  ✅                  │  │
│  │  sees: global    ✅                  │  │
│  │  sees: innerVar  ❌  (no inward)     │  │
│  └──────────────────────────────────────┘  │
│                                            │
│  sees: global    ✅                        │
│  sees: outerVar  ❌  (no inward)           │
│  sees: innerVar  ❌  (no inward)           │
└────────────────────────────────────────────┘
```

## Scope Chain — How JS Actually Finds a Variable

When JS encounters a variable name, it doesn't just look in the current scope.  
It follows a **chain** — searching outward, one scope at a time.

### The lookup process

```
Step 1: Look in current scope
        → Found? Use it. Done.
        → Not found? Go to Step 2.

Step 2: Look in the parent scope
        → Found? Use it. Done.
        → Not found? Go to Step 3.

Step 3: Keep going outward...
        → Eventually reach global scope.
        → Found? Use it. Done.
        → Not found? ReferenceError ❌
```

### Example — watching the chain in action

```js
var a = 1;

function outer() {
  var b = 2;

  function inner() {
    var c = 3;

    console.log(c);  // found in own scope         ✅ stops here
    console.log(b);  // not here → outer → found   ✅ stops here
    console.log(a);  // not here → outer → global  ✅ stops here
    console.log(d);  // not here → outer → global → not found ❌
  }

  inner();
}

outer();
```

### Visualizing the chain

```
inner() scope  →  outer() scope  →  global scope  →  ReferenceError
   c = 3              b = 2             a = 1
   
For variable 'b':
   inner: "do I have b?" NO  →  outer: "do I have b?" YES ✅
   
For variable 'd':
   inner: NO  →  outer: NO  →  global: NO  →  ReferenceError ❌
```

> JS always starts from the **innermost** scope and walks **outward**.  
> It NEVER searches inward or sideways — only outward toward global.

## Lexical Scope — Scope is Set at Write Time, Not Run Time

This is one of the most important concepts in all of JavaScript.

> **Lexical scope** means: a function's scope is determined by **where it is written** in the code — not where it is called from.

```js
var x = "global";

function foo() {
  console.log(x);  // what will this print?
}

function bar() {
  var x = "bar's x";
  foo();           // foo is CALLED from inside bar
}

bar();  // prints: "global" — NOT "bar's x"
```

**Why?** Because `foo` is **written** in global scope. Its scope chain is locked to where it was written — not where it's invoked. When `foo` looks up `x`, it follows its own chain: `foo scope → global scope`. It never looks inside `bar`.

```
foo's scope chain (set at WRITE time, never changes):
   foo scope  →  global scope

bar's x is completely invisible to foo.
Even when foo is called FROM inside bar.
```

> **This is the foundation of closures** (Week 2, Day 3).  
> A function carries its scope with it wherever it goes.  
> Understanding this now makes closures click instantly.

## Variable Shadowing — Inner Variable Hides Outer

When an inner scope declares a variable with the **same name** as an outer scope variable, the inner one **shadows** (hides) the outer one — but only inside that inner scope.

```js
var x = "global";

function outer() {
  var x = "outer";        // new variable — shadows global x

  function inner() {
    var x = "inner";      // new variable — shadows outer x
    console.log(x);       // "inner" ← sees its own x
  }

  inner();
  console.log(x);         // "outer" ← sees its own x
}

outer();
console.log(x);           // "global" ← original untouched
```

**Key point:** Shadowing creates a **new variable** — it does NOT modify the outer one.  
The outer variable still exists, it's just temporarily hidden from the inner scope.

### The common bug — thinking you're updating the outer variable

```js
let count = 0;

function increment() {
  let count = 1;           // BUG: this creates a NEW count, not updating outer
  console.log(count);      // 1
}

increment();
console.log(count);        // 0 ← outer count never changed!
```

Fix: remove the `let` inside the function if you mean to update the outer variable.

```js
let count = 0;

function increment() {
  count = 1;               // ✅ now updating the outer count
}

increment();
console.log(count);        // 1 ✅
```

## Scope + Hoisting — How They Connect

Hoisting always happens **within a scope**.

- `var` hoists to the top of its **function scope** (or global if outside functions)
- `let`/`const` hoist to the top of their **block scope** (into TDZ)

```js
function test() {
  console.log(a);   // undefined — var hoisted to top of function scope
  console.log(b);   // ReferenceError — let hoisted to top of block scope (TDZ)

  var a = 1;
  let b = 2;
}
```

```
FUNCTION SCOPE of test():
┌──────────────────────────────────────┐
│  var a → undefined  (hoisted here)   │
│  let b → TDZ        (hoisted here)   │
│                                      │
│  console.log(a)  → undefined         │
│  console.log(b)  → ReferenceError    │
│  var a = 1       → a is now 1        │
│  let b = 2       → b exits TDZ       │
└──────────────────────────────────────┘
```

## Practical Scope Rules to Remember

**Rule 1:** Always use `let`/`const`. Never `var`. `var`'s function scope causes leaks and bugs.

**Rule 2:** Scope chain goes outward only — inner sees outer, never the reverse.

**Rule 3:** Lexical scope is fixed at write time — calling a function from a different place doesn't change its scope chain.

**Rule 4:** Shadowing creates a new variable — it doesn't update the outer one.

**Rule 5:** Each function call creates a fresh scope — variables inside don't persist between calls (unless closures are involved).

## Output Questions — Work Through Before Reading Answers

**Q1 — Basic scope and shadowing**
```js
var x = 1;
function foo() {
  var x = 2;
  console.log(x);
}
foo();
console.log(x);
```
<details>
<summary>Answer</summary>

`2`, then `1`.  
Two separate `x` variables. Inner `x` shadows the outer one inside `foo`. The outer `x` is completely untouched.
</details>

**Q2 — Scope chain lookup**
```js
var a = 1;
function outer() {
  var b = 2;
  function inner() {
    var c = 3;
    console.log(a + b + c);
  }
  inner();
}
outer();
```
<details>
<summary>Answer</summary>

`6`.  
`c` found in inner scope. `b` found in outer scope via chain. `a` found in global scope via chain. All three add up.
</details>

**Q3 — Outer cannot see inner**
```js
function outer() {
  function inner() {
    var secret = 42;
  }
  inner();
  console.log(secret);
}
outer();
```
<details>
<summary>Answer</summary>

`ReferenceError: secret is not defined`.  
Outer scope cannot look into inner scope. The chain only goes outward, never inward.
</details>

**Q4 — Lexical scope (the key concept)**
```js
var x = "global";

function foo() {
  console.log(x);
}

function bar() {
  var x = "bar";
  foo();
}

bar();
```
<details>
<summary>Answer</summary>

`"global"`.  
`foo` is written in global scope. Its scope chain is locked to where it was written — not where it is called. `foo` looks up `x` in its own chain: `foo scope → global scope`. It never looks in `bar`. This is lexical scope.
</details>

---

**Q5 — var vs let in a block**
```js
function test() {
  if (true) {
    var a = 1;
    let b = 2;
  }
  console.log(a);
  console.log(b);
}
test();
```
<details>
<summary>Answer</summary>

`1` is logged.  
Then `ReferenceError: b is not defined`.  
`var a` is function-scoped — it leaks out of the `if` block. `let b` is block-scoped — it stays inside the `if` block.
</details>

---

**Q6 — Shadowing with let**
```js
let x = "outer";
{
  let x = "inner";
  console.log(x);
}
console.log(x);
```
<details>
<summary>Answer</summary>

`"inner"`, then `"outer"`.  
The inner block creates a new `x` that shadows the outer one. After the block ends, only the outer `x` exists again.
</details>

---

**Q7 — Scope chain with missing variable**
```js
function outer() {
  var a = 1;
  function inner() {
    console.log(a);
    console.log(b);
  }
  inner();
}
outer();
```
<details>
<summary>Answer</summary>

`1` is logged (found in outer via scope chain).  
Then `ReferenceError: b is not defined` — `b` not found in inner, outer, or global scope. Chain exhausted.
</details>

---

**Q8 — The shadowing bug (very common in interviews)**
```js
let count = 0;

function increment() {
  let count = 1;
  console.log(count);
}

increment();
console.log(count);
```
<details>
<summary>Answer</summary>

`1`, then `0`.  
The `let count = 1` inside the function creates a brand new variable — it does NOT update the outer `count`. The outer `count` remains `0`. Classic shadowing bug.
</details>

---

**Q9 — Nested lexical scope**
```js
function makeCounter() {
  var count = 0;

  return function() {
    count++;
    console.log(count);
  };
}

var counter = makeCounter();
counter();
counter();
counter();
```
<details>
<summary>Answer</summary>

`1`, `2`, `3`.  
The inner function is written inside `makeCounter` — so its scope chain includes `makeCounter`'s scope. It can access and modify `count`. Even after `makeCounter()` returns, the inner function carries its scope with it. This is a closure — the full concept is Week 2, Day 3.
</details>

---

**Q10 — var hoisting within function scope**
```js
function test() {
  console.log(x);
  if (false) {
    var x = 5;
  }
  console.log(x);
}
test();
```
<details>
<summary>Answer</summary>

`undefined`, then `undefined`.  
`var x` is hoisted to the top of the function scope during creation phase — so `x` exists and is `undefined` on the first log. The `if (false)` block never runs, so `x` is never assigned. Second log is also `undefined`. Note: the `if` block condition being `false` doesn't stop hoisting — hoisting happens before execution.
</details>

---

**Q11 — Block scope with for loop**
```js
for (let i = 0; i < 3; i++) {
  // nothing
}
console.log(i);

for (var j = 0; j < 3; j++) {
  // nothing
}
console.log(j);
```
<details>
<summary>Answer</summary>

`ReferenceError: i is not defined` — `let` is block-scoped, stays inside the loop.  
`3` — `var j` leaks out to the enclosing function/global scope.
</details>

## What SDE2 Interviewers Expect

**1. Know all three scope types without hesitation**  
Global (everywhere), function (inside function only), block (inside `{}` — only `let`/`const`).  
And the key: `var` only respects function scope, `let`/`const` respect block scope.

**2. Explain the scope chain clearly**  
*"When JS looks up a variable, it starts in the current scope and walks outward — one scope at a time — until it finds it or reaches global and throws a ReferenceError. It never searches inward."*

**3. Explain lexical scope**  
*"Scope is determined by where code is written, not where it's called. A function's scope chain is fixed at definition time. This is why closures work — the function carries its scope wherever it goes."*

**4. Spot the shadowing bug in code**  
Declaring a same-name variable in an inner scope creates a new variable — it does NOT update the outer one. Very common source of real bugs.

**5. Connect scope to hoisting**  
*"Hoisting happens within a scope. `var` hoists to the top of its function scope. `let`/`const` hoist to the top of their block scope, but land in TDZ."*

**6. Know that lexical scope leads to closures (preview)**  
If an interviewer asks about scope, it's often a warmup for closures. End with:  
*"This lexical scoping is exactly what makes closures possible — a function remembers the scope it was created in."*

## The Mental Models to Carry Into the Interview

**Scope = rooms in a house.**  
You can always go out to the hallway, then the living room. You can never enter another bedroom.

**Scope chain = looking through nested boxes outward.**  
Start in your box. Not there? Open the box you're inside. Keep going until global. Still not there? ReferenceError.

**Lexical scope = your scope is set the moment you're written.**  
It doesn't matter who calls you or from where. Your chain is decided at write time, frozen forever.
