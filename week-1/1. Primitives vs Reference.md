## The Two Categories

### Primitives (7 types)
Stored **by value**. Copied on assignment. Immutable — operations return new values, never modify the original.

```
string    number    boolean
null      undefined symbol
bigint
```

### Reference Types
Stored **by reference** (memory address). Shared on assignment. Mutable in place.

```
object    // {}
array     // []
function  // () => {}
```

---

## Memory Model

Primitives live entirely on the **stack**.  
References split: the variable on the stack holds an *address* pointing to actual data on the **heap**.

### Primitive — independent copies

```js
let a = 10;
let b = a;  // b gets a COPY of the value
b = 20;
console.log(a); // 10 ← unchanged
```

```
Stack
┌─────────────────┐
│  a  │   10      │  ← independent slots
│  b  │   20      │
└─────────────────┘
```

Changing `b` has zero effect on `a`.

---

### Reference — shared pointer

```js
let a = { name: "Alice" };
let b = a;  // b gets the ADDRESS, not a copy
b.name = "Bob";
console.log(a.name); // "Bob" ← a was affected!
```

```
Stack                    Heap
┌──────────────┐         ┌─────────────────────┐
│  a  │ 0x001 │────────▶│  0x001              │
│  b  │ 0x001 │────────▶│  { name: "Bob" }    │
└──────────────┘         └─────────────────────┘
```

One object, two pointers. Mutating through `b` is visible via `a`.

## Function Arguments — The Full Mental Model

### The Real Rule

JavaScript **always** passes arguments by value.

- For primitives → the value is the actual data
- For objects → the value is a **copy of the reference (memory address)**

> "Objects are not passed by reference. A copy of the reference is passed."

This means JavaScript does **not** support true pass-by-reference. When a function is called, a new variable is created in the function scope and receives a copy of the argument's value. For objects, that copied value is the memory address — which is why mutations are visible outside, but reassignments are not.

---

### Case 1: Primitive — copy of value

```js
function change(x) {
  x = 100;
}

let num = 10;
change(num);
console.log(num); // 10 — NOT changed
```

**Memory flow:**

Before call:
```
Stack
┌──────────────┐
│ num │  10    │
└──────────────┘
```

During call:
```
Stack
┌──────────────┐
│ num │  10    │
│ x   │  10    │  ← copy of value
└──────────────┘
```

After modification inside function:
```
Stack
┌──────────────┐
│ num │  10    │  ← unchanged
│ x   │ 100    │  ← only x changed
└──────────────┘
```

`num` is unchanged because `x` is an independent copy.

---

### Case 2: Object mutation — visible outside

```js
function change(obj) {
  obj.name = "Bob";
}

let user = { name: "Alice" };
change(user);
console.log(user.name); // "Bob" ← WAS changed
```

**Memory flow:**

Before call:
```
Stack                    Heap
┌──────────────┐         ┌─────────────────────┐
│ user │ 0x001 │────────▶│ { name: "Alice" }   │
└──────────────┘         └─────────────────────┘
```

During call (`obj` receives a copy of the reference `0x001`):
```
Stack                    Heap
┌──────────────┐         ┌─────────────────────┐
│ user │ 0x001 │────────▶│ { name: "Alice" }   │
│ obj  │ 0x001 │────────▶│                     │
└──────────────┘         └─────────────────────┘
```

After mutation:
```
Stack                    Heap
┌──────────────┐         ┌─────────────────────┐
│ user │ 0x001 │────────▶│ { name: "Bob" }     │
│ obj  │ 0x001 │────────▶│                     │
└──────────────┘         └─────────────────────┘
```

Both point to the same object → mutation is visible outside.

---

### Case 3: Reassignment inside function — does NOT affect original

```js
function change(obj) {
  obj = { name: "Bob" }; // points local var to a new object
}

let user = { name: "Alice" };
change(user);
console.log(user.name); // "Alice" ← NOT changed
```

**Memory flow:**

Before reassignment:
```
Stack                    Heap
┌──────────────┐         ┌─────────────────────┐
│ user │ 0x001 │────────▶│ { name: "Alice" }   │
│ obj  │ 0x001 │────────▶│                     │
└──────────────┘         └─────────────────────┘
```

After reassignment:
```
Stack                    Heap
┌──────────────┐         ┌─────────────────────┐
│ user │ 0x001 │────────▶│ { name: "Alice" }   │
│ obj  │ 0x002 │────────▶│ { name: "Bob" }     │
└──────────────┘         └─────────────────────┘
```

`obj` now points to a new object. `user` still points to the original.

---

### Mutation vs Reassignment — Key Distinction

```js
obj.name = "Bob"  // mutation    → affects the original object
obj = {}          // reassignment → does NOT affect the original
```

> **Real-world relevance:** This is why mutating function arguments is considered bad practice — callers don't expect side effects. Prefer returning a new object or using immutable patterns.

---

### Final Mental Model

- JavaScript is always pass-by-value
- For objects, that value is a reference (memory address)
- Functions receive a copy of that value
- Mutation works because both copies reference the same heap object
- Reassignment only changes the local copy — the caller's variable is untouched

## Equality Comparison

### Primitives → compared by value

```js
let a = "hello";
let b = "hello";
a === b; // true — same characters = same value
```

### References → compared by address (memory location)

```js
let a = { x: 1 };
let b = { x: 1 };
a === b; // false ← different objects in memory

let c = a;
a === c; // true ← same reference
```

Two objects that look identical are **not equal** unless they're literally the same object in memory.

---

## Immutability of Primitives

Primitives cannot be mutated. Operations always return a new value.

```js
let str = "hello";
str.toUpperCase(); // returns "HELLO" but...
console.log(str);  // "hello" ← original unchanged
```

String methods always return a *new* string. The original is never touched.

---

## `const` — The Common Misconception

`const` prevents **reassignment**, not **mutation**.

```js
const obj = { name: "Alice" };
obj.name = "Bob";      // ✅ OK — mutating the object
obj = { name: "Bob" }; // ❌ TypeError — reassigning the variable

const arr = [1, 2, 3];
arr.push(4);   // ✅ OK
arr = [1, 2];  // ❌ TypeError
```

> If you need true immutability, use `Object.freeze()` — but note it's shallow only.

## `typeof` Gotchas

```js
typeof "hello"      // "string"
typeof 42           // "number"
typeof true         // "boolean"
typeof undefined    // "undefined"
typeof null         // "object"    ← famous JS bug, never fixed
typeof []           // "object"    ← not "array"
typeof {}           // "object"
typeof function(){} // "function"  ← exception to the rule

Array.isArray([])   // true ← correct way to check for arrays
```

`typeof null === "object"` is a legacy bug from JS's early days. It was never fixed to avoid breaking the web.

## Autoboxing — How Primitives Have Methods

**Interview question:** *"If primitives have no methods, how does `"hello".toUpperCase()` work?"*

**Answer:** JS temporarily wraps the primitive in its object wrapper (`String`, `Number`, `Boolean`), calls the method, then immediately discards the wrapper. This is called **autoboxing**.

```js
let a = "hello"             // primitive
let b = new String("hello") // object wrapper (never use this in practice)

typeof a // "string"
typeof b // "object"
a === b  // false ← different types
```

You never use `new String()`, `new Number()`, etc. in real code. But understanding autoboxing answers *why* primitives appear to have methods.

**Q1 — Basic primitive copy**
```js
let a = 5;
let b = a;
b++;
console.log(a); // ?
console.log(b); // ?
```
<details>
<summary>Answer</summary>

`a → 5`, `b → 6`  
Primitives are copied. `b++` only affects `b`.
</details>

---

**Q2 — Array mutation through reference**
```js
let a = [1, 2, 3];
let b = a;
b.push(4);
console.log(a); // ?
```
<details>
<summary>Answer</summary>

`[1, 2, 3, 4]`  
`b` and `a` point to the same array. `push()` mutates it in place.
</details>

---

**Q3 — Reassignment breaks the link**
```js
let a = [1, 2, 3];
let b = a;
b = [1, 2, 3, 4]; // reassignment
console.log(a); // ?
```
<details>
<summary>Answer</summary>

`[1, 2, 3]`  
Reassigning `b` points it to a new array. `a` is unaffected.
</details>

---

**Q4 — Object equality**
```js
let a = { x: 1 };
let b = { x: 1 };
console.log(a == b);  // ?
console.log(a === b); // ?
```
<details>
<summary>Answer</summary>

Both `false`  
Different objects in memory regardless of which equality operator is used.
</details>

---

**Q5 — The hard one: mutation then reassignment in a function**
```js
function update(obj) {
  obj.x = 10;        // mutates the original
  obj = { x: 20 };   // reassigns local var only
}

let a = { x: 1 };
update(a);
console.log(a.x); // ?
```
<details>
<summary>Answer</summary>

`10`  
`obj.x = 10` mutated the original object **before** `obj` was reassigned. The reassignment only affects the local variable inside the function. The mutation already happened.
</details>

---

**Q6 — `const` with object**
```js
const user = { name: "Alice" };
user.name = "Bob";
console.log(user.name); // ?
```
<details>
<summary>Answer</summary>

`"Bob"`  
`const` prevents reassignment of the **variable**, not mutation of the object it points to.
</details>

---

**Q7 — `typeof` null**
```js
console.log(typeof null);       // ?
console.log(typeof []);          // ?
console.log(Array.isArray([]));  // ?
```
<details>
<summary>Answer</summary>

`"object"`, `"object"`, `true`  
`typeof null` is a legacy JS bug. `typeof []` returns `"object"` too. Always use `Array.isArray()` to check for arrays.
</details>
